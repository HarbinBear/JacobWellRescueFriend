### P1：角色控制——手动挡模式 🔴 高优先

**问题描述**：
当前只有虚拟摇杆（自动挡）移动方式。需要新增"手动挡"模式：用户搓一次屏幕代表一次移动指令。

**设计要点**：
- 从下往上搓 → 角色向下移动（推水方向与移动方向相反）
- 从左往右搓 → 角色向左移动
- 手不动 → 角色停止移动
- 转向不灵敏：比如掉头需要搓 3 次才能完成 180° 转向，前进方向的调整亦不是一蹴而就。
- 有原地掉头动作
- 操作写实感：角色移动与玩家手势一致
- 支持两个手指同时搓，对应了双手轮流搓的场景，就像是潜水员双脚在交替划水。

**任务拆分**：
- [x] T1.1 在 `input.ts` 中新增手动挡输入识别（滑动手势检测、方向计算）
- [x] T1.2 在 `config.ts` 中新增手动挡相关参数（单次推进距离、转向步长、阻尼等）
- [x] T1.3 在 `state.ts` 中新增手动挡状态字段（当前模式、滑动队列等）
- [x] T1.4 在 `Logic.ts` 中新增手动挡移动逻辑（脉冲式推进、渐进转向、水阻衰减）
- [x] T1.5 在 GM 面板中新增自动挡/手动挡切换开关

### P3：岩石生成一致性 🔴 高优先

**问题描述**：

**设计要点**：

## 三、开发迭代计划

### 迭代原则

1. **每次迭代优先处理高优先级需求**
2. **有依赖关系的任务按依赖顺序排列**
3. **每次迭代控制在合理范围内，确保可交付**
4. **每次迭代完成后更新本文档**

### 依赖关系图

```
P2（角色表现）──→ P1（手动挡，需要划水动画）
P4（水体尘埃）──→ P6（VPL 连续化，光照基础改进）
P1（手动挡）──→ P9（划水音效依赖移动模式）

P3（岩石一致性）──→ P5（标记系统，标记位置依赖准确的岩石表面）
```

### 建议迭代顺序

#### 第一轮迭代：角色基础（P2 + P3）
- **P2**：重绘潜水员，建立身体结构和基础动画

- 理由：角色和岩石是最基础的视觉元素，后续所有系统都依赖它们

#### 第三轮迭代：水体与光照（P4 + P6）
- **P4**：水中悬浮尘埃
- **P6**：VPL 连续反射面
- 理由：两者都涉及光照系统改进，适合一起做

#### 第四轮迭代：相机 + 浅水区（P7 + P8）
- **P7**：弹簧臂相机 + 水中摇曳
- **P8**：阳光系统 + 水面焦散
- 理由：都是表现层增强，不影响核心玩法

- **P9**：音频系统
- **P10**：生命系统增强
- 理由：低优先级，且 P10 依赖 P9

## 四、已完成工作记录

### P1：手动挡模式（2026-04-05 ~ 2026-04-08）

**设计思路**：
手动挡模式将连续摇杆输入替换为搓屏实时驱动。经过 V1~V5 五次迭代，最终采用「推力沿输入方向 + 各向异性水阻 + 身体被动跟随速度方向」的物理模型。

**迭代历程**：
- V1（脉冲模型）：touchEnd 产生脉冲，逻辑层消费 → 松手才动，转向角度太小
- V2（实时驱动）：touchMove 实时更新，从起始点算方向 → 手指不动也一直前进
- V3（瞬时速度）：帧间位移驱动，手指停住无推力 → 统一水阻，无方向性
- V4（各向异性水阻）：前向/侧向不同水阻 → 但推力沿身体朝向，向后搓却往前冲
- V5（最终版）：推力沿输入方向施加，身体被动跟随速度方向 → 物理正确

**核心机制（V5 最终版）**：
1. **推力沿输入方向施加**：搓哪个方向，力就往哪个方向推。不经过身体朝向中转，不会出现"向后搓却往前冲"
2. **身体朝向被动跟随速度方向**：水中流线型身体自然对齐运动方向（`bodyAlignRate=0.12`），速度低于 `bodyAlignMinSpeed` 时不跟随（避免静止抖动）
3. **各向异性水阻**：前向阻力小（`dragForward=0.95`，流线型）、侧向阻力大（`dragLateral=0.82`）。这是身体对齐速度方向的物理原因
4. **掉头过程**：向后搓 → 推力反向 → 先减速 → 速度反转 → 身体跟着转过来。整个过程完全由物理驱动，没有任何"转向速率"硬限制
5. **搓速映射推力**：帧间位移越大（搓得越快），推力越大（`thrustBase + strokeStrength * thrustSwipeScale`），上限 `thrustMax`
6. **限速**：最大速度 `maxSpeed=5.0`
7. **双指支持**：最多同时跟踪2个触点，取帧间位移最大的触点作为本帧划水源
8. **prev/curr 分离**：input 层只更新 curr，逻辑层每帧消费后推进 prev，确保手指不动时 frameDist=0
9. **调试辅助线**：GM 面板可开启，显示身体朝向（绿色）、速度向量（黄色）、输入方向（红色虚线）、前向/侧向速度分量与偏差角度数值

### P3：岩石生成一致性（2026-04-04）

**问题根因**：
1. 碰撞检测硬编码 `+10` padding，但渲染和光照不加 padding → 碰撞范围比视觉范围大
2. 光照射线遍历 `map` 格子，每格只有一个 wall 对象；迷宫额外装饰圆只在 `walls` 数组中 → 额外圆不参与光照遮挡
3. 三个模式（主线/竞技场/迷宫）的碰撞 padding 不统一

**修改内容**：
- `config.ts`：新增 `playerRadius: 10`（主线/竞技场）和 `maze.playerRadius: 12`（迷宫），语义清晰
- `Logic.ts`：`checkCollision()` 和 `checkMazeCollision()` 改用 `CONFIG.playerRadius` / `CONFIG.maze.playerRadius`，不再硬编码
- `FishEnemy.ts`：`checkCollisionLocal()` 同步使用 `CONFIG.playerRadius`
- `RenderLight.ts`：
  - `getActiveMapContext()` 新增返回 `walls` 数组
  - `getLightPolygon()` 除遍历 map 格子外，还遍历 walls 数组中的额外圆（去重），确保迷宫装饰圆也参与光照遮挡
  - `isLineOfSight()` 同步检测额外圆

**一致性保证**：
- 渲染 = `wall.r`（视觉半径）
- 光照遮挡 = `wall.r`（射线无体积，精确匹配）
- 碰撞 = `wall.r + playerRadius`（玩家有体积，不能贴墙）
- 三者使用同一个 wall 对象数据源，偏移完全一致

**迷宫额外圆碰撞修复（2026-04-05）**：
- 发现迷宫模式每个边缘格子会额外生成 1-2 个装饰圆（在 `mazeWalls` 中但不在 `mazeMap` 中），这些圆被渲染了但碰撞检测看不到它们
- 玩家可以穿过这些额外圆，直到碰到基础 wall 才停下——这就是"穿进岩石里再碰撞"的根因
- 修改 `map.ts`：额外圆生成时同时挂到基础 wall 的 `extras` 数组上
- 修改 `Logic.ts`：`checkMazeCollision()` 检测到 wall 对象时，同时遍历 `cell.extras` 中的额外圆
- 所有碰撞检测搜索范围统一为 5x5（`r±2, c±2`），与 `getNearestWallDist()` 一致

### P4：悬浮尘埃系统（2026-04-05）

**技术方案**：
不使用传统粒子系统（生命周期管理），而是采用**空间哈希确定性采样**：
- 将世界空间划分为固定大小的格子（默认80px）
- 每个格子内用伪随机哈希确定性生成固定数量的尘埃位置
- 只遍历视口内的格子，不存储全局粒子列表
- 尘埃位置永久固定在世界坐标中，相机移动时自然产生视差参照效果

**双层渲染架构**：
1. **暗色层**（光照遮罩之前绘制）：微弱灰白色尘埃，作为移动参照物
2. **亮色层**（光照遮罩之后绘制）：被手电照亮的尘埃产生暖白色散射光晕

**尘埃动态**：
- 多频正弦叠加模拟布朗运动漂移
- 每个尘埃有独立的漂移相位（由哈希确定性生成）
- 漂移幅度和速度可通过 GM 面板实时调整

**手电交互**：
- 亮色层检测尘埃是否在手电光锥内（角度 + 距离判定）
- 距离平方衰减 + 角度边缘渐变
- 被照亮的尘埃产生散射光晕（较大的柔和光圈）+ 核心亮点

**性能优化**：
- 只遍历视口内格子（视口裁剪）
- 岩石内部尘埃严格剔除（检查当前格子及 3x3 邻域的所有 wall 对象和 extras 装饰圆，确保尘埃完全在水中）
- 深度密度缩放（浅水区少、深水区多）
- 亮色层额外做光锥范围预剔除

**修改文件**：
- `src/render/DustMotes.ts`（新建）：尘埃系统核心模块
- `src/core/config.ts`：新增 `dust` 配置子对象（14个参数）
- `src/render/Render.ts`：在角色绘制后插入暗色层，在光照后插入亮色层
- `src/gm/GMConfig.ts`：新增「尘埃」Tab 页签

### P2：角色表现设计稿（2026-04-08）

**设计结论**：
- 手动挡输入的主可视化应是**潜水员划水动作 + 姿态修正 + 局部水流扰动**，而不是强 UI 化的手势轨迹
- 潜水员俯视角轮廓采用**前圆、中厚、后分叉**结构，重点强化**头部面镜、背部气瓶、双蛙鞋**三个识别锚点
- 动画语义拆为四类：**漂浮待机、划水推进、滑行、掉头修正**，不做单一持续游泳循环
- 反向搓屏的视觉表现应是**刹车 + 拧身 + 掉头**，而不是简单倒车
- 双指搓屏应尽量表现为**双侧交替发力**，让角色看起来像真实连续划水

**文档落点**：
- `design/04-interaction-and-engineering.md`：新增“潜水员角色表现与手动挡输入可视化设计稿”完整章节，作为 P2 与 P1.6 的正式设计依据

**实现细调（2026-04-08 夜）**：
- 去掉了潜水员四肢和蛙鞋上的细浅蓝辅线，避免小尺寸下像噪点或蚊子腿
- 去掉了头部高饱和黄色面镜点，改成更克制的灰蓝色面镜
- 放慢了手臂待机摆动、腿部滑行摆动和整次划水脉冲的默认节奏
- `manualDrive.strokeDuration` 改为统一控制单次划水动作时长
- `config.ts` 新增 `diver` 配置组，`GMConfig.ts` 新增“角色”Tab，可直接调手臂频率/幅度、腿部频率/幅度、漂浮速度和蛙鞋开合

**实现细调（2026-04-08 深夜第二轮）**：
- 去掉了头部灰色小点和手部肉色小点，避免在小尺寸下出现多余视觉锚点
- 手臂改为只做轻微待机摆动和转向修正，不再承担主划水推进
- 腿部主动作改为上下踢水，不再使用左右扫腿式推进语义
- 手动挡输入从“本帧只取最大触点”重构为“逐触点跟踪、逐触点单次消费”
- 每个触点从按下到松开只会驱动一次完整踢水；不松手不会重复触发第二次
- 输入向量会拆成前向分量与侧向分量：前向分量驱动推进踢水，侧向分量驱动手脚一起参与的转向修正，并在线性叠加后共同进入角色表现
- 左右侧输入现在会直接映射到左右腿，可精确匹配任意左右输入节奏；双侧同时输入时允许双腿同时发力
- `config.ts` 和 `GMConfig.ts` 已同步改成新的参数语义（单次踢水距离、转向速率、腿部伸展/回收、转向腿偏移等）

**实现细调（2026-04-09 凌晨第三轮）**：
- 手动挡的“前进和转向太慢”问题已直接上调默认强度：基础推进、速度加成、基础转向和最大速度都重新收口
- 单次输入不再只在前半段有效；改为整段 `effectiveDistance` 有效行程持续驱动移动与踢水表现，并提供明确参数入口可直接拉长行程
- 同一段输入距离下，输入速度现在会影响推进和转向力度；搓得更快会更有力，搓得慢则更柔和
- 左右腿判定已从“屏幕左右”修正为“角色局部左右”，按当前角色朝向实时计算触点位于身体左侧还是右侧
