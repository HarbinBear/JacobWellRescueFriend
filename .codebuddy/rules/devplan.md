### P1：角色控制——手动挡模式 🔴 高优先

**问题描述**：
当前只有虚拟摇杆（自动挡）移动方式。需要新增"手动挡"模式：用户搓一次屏幕代表一次移动指令。

**设计要点**：
- 从下往上搓 → 角色向下移动（推水方向与移动方向相反）
- 从左往右搓 → 角色向左移动
- 手不动 → 角色停止移动
- 转向不灵敏：比如掉头需要搓 3 次才能完成 180° 转向，前进方向的调整亦不是一蹴而就。
- 有原地掉头动作
- 操作写实感：角色移动与玩家手势一致
- 支持两个手指同时搓，对应了双手轮流搓的场景，就像是潜水员双脚在交替划水。

**任务拆分**：
- [x] T1.1 在 `input.ts` 中新增手动挡输入识别（滑动手势检测、方向计算）
- [x] T1.2 在 `config.ts` 中新增手动挡相关参数（单次推进距离、转向步长、阻尼等）
- [x] T1.3 在 `state.ts` 中新增手动挡状态字段（当前模式、滑动队列等）
- [x] T1.4 在 `Logic.ts` 中新增手动挡移动逻辑（脉冲式推进、渐进转向、水阻衰减）
- [x] T1.5 在 GM 面板中新增自动挡/手动挡切换开关

- [x] T1.7 第一版联调与参数收口
- [ ] T1.8 真机测试与手感细调

### P3：岩石生成一致性 🔴 高优先

1. **每次迭代优先处理高优先级需求**
2. **有依赖关系的任务按依赖顺序排列**
3. **每次迭代控制在合理范围内，确保可交付**
4. **每次迭代完成后更新本文档**

```
P2（角色表现）──→ P1（手动挡，需要划水动画）
P4（水体尘埃）──→ P6（VPL 连续化，光照基础改进）

P3（岩石一致性）──→ P5（标记系统，标记位置依赖准确的岩石表面）
```

### 建议迭代顺序

#### 第一轮迭代：角色基础（P2 + P3）
- **P2**：重绘潜水员，建立身体结构和基础动画

- 理由：角色和岩石是最基础的视觉元素，后续所有系统都依赖它们

- **P1**：手动挡移动模式（依赖 P2 的划水动画）

- 理由：控制和交互是核心玩法，需要尽早验证手感

#### 第三轮迭代：水体与光照（P4 + P6）
- **P4**：水中悬浮尘埃
- **P6**：VPL 连续反射面
- 理由：两者都涉及光照系统改进，适合一起做

#### 第四轮迭代：相机 + 浅水区（P7 + P8）
- **P7**：弹簧臂相机 + 水中摇曳

- 理由：都是表现层增强，不影响核心玩法

- **P9**：音频系统
- **P10**：生命系统增强
- 理由：低优先级，且 P10 依赖 P9

## 四、已完成工作记录

### P1：手动挡模式（2026-04-05 ~ 2026-04-08）

**设计思路**：
手动挡模式将连续摇杆输入替换为搓屏实时驱动。经过 V1~V5 五次迭代，最终采用「推力沿输入方向 + 各向异性水阻 + 身体被动跟随速度方向」的物理模型。

**迭代历程**：
- V1（脉冲模型）：touchEnd 产生脉冲，逻辑层消费 → 松手才动，转向角度太小
- V2（实时驱动）：touchMove 实时更新，从起始点算方向 → 手指不动也一直前进
- V3（瞬时速度）：帧间位移驱动，手指停住无推力 → 统一水阻，无方向性
- V4（各向异性水阻）：前向/侧向不同水阻 → 但推力沿身体朝向，向后搓却往前冲
- V5（最终版）：推力沿输入方向施加，身体被动跟随速度方向 → 物理正确

**核心机制（V5 最终版）**：
1. **推力沿输入方向施加**：搓哪个方向，力就往哪个方向推。不经过身体朝向中转，不会出现"向后搓却往前冲"
2. **身体朝向被动跟随速度方向**：水中流线型身体自然对齐运动方向（`bodyAlignRate=0.12`），速度低于 `bodyAlignMinSpeed` 时不跟随（避免静止抖动）
3. **各向异性水阻**：前向阻力小（`dragForward=0.95`，流线型）、侧向阻力大（`dragLateral=0.82`）。这是身体对齐速度方向的物理原因
4. **掉头过程**：向后搓 → 推力反向 → 先减速 → 速度反转 → 身体跟着转过来。整个过程完全由物理驱动，没有任何"转向速率"硬限制
5. **搓速映射推力**：帧间位移越大（搓得越快），推力越大（`thrustBase + strokeStrength * thrustSwipeScale`），上限 `thrustMax`
6. **限速**：最大速度 `maxSpeed=5.0`
7. **双指支持**：最多同时跟踪2个触点，取帧间位移最大的触点作为本帧划水源
8. **prev/curr 分离**：input 层只更新 curr，逻辑层每帧消费后推进 prev，确保手指不动时 frameDist=0

### P8：浅水区渲染——阳光系统（2026-04-11 ~ 2026-04-12）

**设计思路**：
迷宫模式浅水区需要从水面到深处有完整的视觉表现：天空、水面波浪、阳光光柱、水体色调、焦散光斑，以及最关键的——全局环境光遮罩从水面到深处的连续平滑过渡。

**已完成的核心功能**：

1. **天空连续化（T8.1）**：从天空顶到浅水区底全程不透明渐变，用 20 个均匀色停覆盖全程，彻底消除色带。天空部分分三段（天空顶→天空中→水面色），水下部分用幂函数让前段变化快、后段平稳
2. **阳光平行光柱（T8.2）**：从水面向下照射的宽幅光束，有独立摇曳相位、深度扩散和衰减。支持配置光柱数量、宽度、长度、强度、间距、摇曳速度/幅度、入射角度、颜色
3. **水面焦散效果（T8.4）**：伪随机网格生成焦散光斑，随时间缓慢移动和变形，浅水因子控制衰减
4. **水面反光带（T8.5）**：水面处柔和高光渐变带 + 双层波浪（后层暗慢 + 前层亮快）
5. **水体色调叠加（drawMazeShallowWaterTint）**：大垂直渐变覆盖整个浅水区，避免逐格子绘制产生色带
6. **丁达尔光柱**：细散射光补充阳光平行光的间隙
7. **GM 面板浅水区 Tab（T8.6）**：18+ 个可调参数

**环境光遮罩系统迭代历程**：

环境光遮罩（`getMazeShallowMaskAlpha`）经历了多次迭代：

- **V1（pow(factor, 0.3) + smoothstep 混合）**：用 `pow(factor, 0.3)` 做衰减，但在 factor→0 时值仍然很高（`pow(0.01, 0.3) = 0.5`），导致浅水区底部 ambient 还有 0.47，然后突然跳到 0.01。后来加的 smoothstep 混合只覆盖底部 15%，不够彻底
- **V2（完全重写，单一连续幂函数曲线）**：用一条从水面到深处完全连续的曲线过渡，没有任何分段拼接。`t = (playerY - surfaceY) / depth`，`darkProgress = pow(t, adjustedExp)`，`ambient = ambientMax + (ambientMin - ambientMax) * darkProgress`。通过 `maskCurveExp` 和 `maskMidPoint` 两个参数控制曲线形状

**其他修复**：
- 水面位置统一：表现上的水面和实际能游到的水面对齐（`waterSurfaceY` 参数）
- 去掉了不规则形状岩石，只保留圆形岩石
- 去掉了水体中的背景圆形和水中线条装饰
- 去掉了岩石反光效果（效果不好，代码清理）
- 浅水区到深水区的背景颜色断层修复
- 岩石颜色随深度连续变暗（不再有突变）

## 六、注意事项

1. 每次迭代完成后必须运行 `npm run typecheck` 确认无 TypeScript 报错
2. 涉及 GM 面板的改动，新增参数统一在 `GMConfig.ts` 的 `TABS` 中添加

5. **修改 `.glsl` 源文件后，必须运行 `node scripts/buildShaders.js` 重新生成 `.glsl.ts`**，否则运行时 shader 仍是旧版本
