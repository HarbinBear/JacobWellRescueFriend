# 设计文档分卷 03：世界结构与核心玩法系统

## 本卷用途

本卷主要回答下面这些问题：

- 地图为什么是现在这种结构
- 区域和地标在设计上承担什么作用
- 玩家为什么会持续感到压力
- 绳索、手电、氧气、凶猛鱼分别服务什么体验
- NPC 在主线中为什么重要

如果任务与地图、地标、生存压力、敌人、绳索有关，应优先阅读本卷。

---

## 一、地图系统设计

### 1.1 主线地图基础规格

主线地图是固定洞穴结构，而非程序化随机图。

参数：

- 地图尺寸：80 列 × 150 行
- 单格大小：40 × 40 像素
- 总尺寸：3200 × 6000 像素

这种纵向长地图非常适合表达“向下深入”的心理感受。

### 1.2 地图数据来源

地图并非纯手写数组，而是通过素材图片处理生成：

- 原始视觉资源：`texture/Well.png`
- 处理脚本：`scripts/processMap.py`
- 产出数据：`src/world/mapData.ts`

这意味着地图编辑工作流更偏向：

- 先画轮廓或区域图
- 再由脚本转成游戏内可用数据

优点：

- 对关卡形状编辑直观
- 便于迭代整体洞穴结构
- 适合项目这种地形主导型玩法

### 1.3 地图格子语义

项目中至少有以下几种格子或碰撞语义：

- `0`：水域，可通行
- `1`：生成过程中的岩石标记，后续会转换
- `wall 对象`：边缘可视岩石，用于渲染圆润墙面与碰撞
- `2`：内部实体岩石，主要服务碰撞，通常不作为圆形边缘渲染

这个设计体现出：

- **渲染墙体**和**碰撞墙体**不是完全同一层抽象。
- 地图最终不是纯数字数组，而是混合数据结构。

### 1.4 区域划分 `state.zones`

区域划分不是装饰性命名，而是用来帮助理解：

- 当前玩家位于哪一段叙事空间
- 哪些剧情应在附近发生
- 哪些路径是主路、支路、死路

现有区域包括：

| 区域名 | 设计语义 |
|--------|----------|
| `entrance` | 水面入口与安全感残留区域 |
| `chamber1` | 早期探索区，较宽，作为进入深处前的缓冲 |
| `suit_tunnel` | 导向潜水服地标的狭窄通道 |
| `chamber2` | 更深入、更偏移的中段洞室 |
| `junction` | 路径决策区，连接多个方向 |
| `dead_end` | 用于误导、恐惧与 NPC 行动演出的假终点 |
| `chamber3` | 深层剧情区，承接第三关恐怖事件 |
| `story_tunnel` | 极窄、专门为剧情压迫感服务的通道 |
| `chamber4` | 终局潜在区域，当前开发中 |

### 1.5 关键地标 `state.landmarks`

地标是剧情与地图绑定的核心纽带。

| 地标 | 用途 |
|------|------|
| `entrance` | 入口出生点，通常自动扫描确定 |
| `suit` | 废弃潜水服位置，触发环境叙事 |
| `tunnelEntry` | 缝隙入口，第一关阻挡、第二关进入 |
| `tunnelEnd` | 玩家被卡住位置，濒死剧情核心点 |
| `deadEndDeep` | 假烟囱深处，服务第二关 NPC 行为 |
| `chamber12Junction` | 第一二洞室连接点，第三关灯损坏触发 |
| `chamber23Junction` | 二三洞室连接点，恐怖事件关键区域 |

### 1.6 地图设计意图

从区域与地标分布可以看出，主线地图并不是“大世界开放探索”，而是：

- 有一定自由度
- 但核心剧情路线上高度引导
- 关键事件强依赖具体坐标
- 空间结构服务情绪爬升

因此后续改图时要极其注意：

- 不要轻易改坏地标相对关系
- 不要让关键剧情点之间的路程节奏失真
- 不要把本应狭窄压迫的路径改成宽阔安全区

## 二、竞技场地图设计

### 2.1 竞技场基础规格

竞技场是独立正方形场地：

- 2000 × 2000 像素
- 格子大小 40 × 40
- 外围存在厚岩石墙

### 2.2 障碍物与空间目标

内部会随机生成矩形障碍物，且会避免离出生点太近。

作用：

- 提供绕行与拉扯空间
- 避免纯空场导致敌人行为单调
- 让玩家在被追逐时有可利用地形
- 让光照与近战攻击产生空间选择

### 2.3 竞技场设计语义

与主线不同，竞技场地图不需要讲故事，它要保证的是：

- 出生安全
- 敌人生成合理
- 路径可周旋
- 小地图信息清晰
- 节奏适合反复开局

因此竞技场全图默认已探索，是一个明确的“玩法场”而不是“叙事场”。

## 三、玩家核心生存系统

### 3.1 移动与朝向

玩家在俯视角下移动，朝向会影响：

- 手电照射方向
- 攻击方向
- 角色绘制方向
- 与敌人或场景的互动感受

在这种题材中，“朝向”不仅是动作参数，也是感知参数。

### 3.2 氧气系统

氧气是项目最基础的生存资源。

它承担的设计作用包括：

- 限制探索深度与停留时间
- 强迫玩家考虑回程
- 在狭窄地带制造焦虑
- 为濒死演出提供硬逻辑支撑

主线中氧气是压力来源，竞技场中则被解除，用于让战斗模式更纯粹。

### 3.3 氮气或深潜压力

虽然项目主打氧气管理，但氮气或深潜相关状态同样服务于“潜水不是简单血条”的概念。

它强调：

- 深度不是纯视觉变化，而是系统性风险变化。
- 玩家在深处承受的不仅是空间威胁，还有身体负担。

### 3.4 手电系统

手电是整个项目中最关键的感知工具之一，兼具：

- 照明
- 导航
- 氛围塑造
- 危险揭示
- 敌人威慑

它并不是简单的画面特效，而是直接影响玩法与情绪的主系统。

### 3.5 受伤、濒死与失败表现

项目中的失败表现通常会通过以下内容组合体现：

- 红屏叠加
- 屏幕震动
- 视野收缩
- 控制权下降
- 失败提示文本

这使得失败不只是一个布尔结果，而是一个逐步逼近的过程。

## 四、引导绳系统设计

### 4.1 引导绳的题材意义

洞穴潜水中的引导绳非常关键，因此它在本项目里既有真实性，也有玩法价值。

它的意义在于：

- 体现专业潜水语境
- 帮助玩家建立路线感
- 在视觉上强化“我来过这里”
- 让深处探索更有仪式感

### 4.2 交互方式

当前绳索交互不是即时点击，而是结合：

- 靠近墙体锚点
- 玩家保持相对静止
- UI 提示出现
- 长按完成起始或结束固定

这种交互方式的设计价值：

- 防止高速移动时误触
- 让“布绳”显得更像一个实际操作动作
- 让锚定行为具有明确意图感

### 4.3 锚点选择规则

绳索系统会在玩家附近寻找合适墙体，并把绳端固定在墙体表面锚点，而不是墙心。

这意味着系统关心的是：

- 视觉合理性
- 绳索与墙体的贴合感
- 路径避障后的可读性

### 4.4 绳索路径

绳索不是简单直线，而会构建一条避开障碍的路径。

这很重要，因为洞穴地图大量存在狭窄墙体结构。若直接画直线，会造成：

- 穿墙
- 视觉不可信
- 玩家误判路线

因此引导绳实际上也是一个轻量路径生成系统的表现层应用。

并且有一条很重要的体验约束：

- 玩家在结束布线前看到的预览绕线路径，完成后必须保持一致；不能出现“预览会绕岩，最终却变直线”的违和感

### 4.5 绳索状态

从设计语义上看，绳索有几种关键状态：

- **未铺设**：等待起点锚定
- **进行中**：已固定起点，尾端跟随玩家
- **收紧中**：准备固定终点时的过渡状态
- **完成**：两端固定，形成稳定引导线

### 4.6 绳索的体验作用

引导绳除了“找路”外，还有几个额外效果：

- 提升洞穴探索的专业感
- 让玩家主动建立安全感
- 让深处场景更有可追溯性
- 为画面增加空间层次

## 五、凶猛鱼与战斗系统设计

### 5.1 凶猛鱼的定位

凶猛鱼不是传统小怪，更接近一种：

- 水下危险生物
- 惊吓与压迫装置
- 带有行为模式的掠食者

在主线里，它服务氛围与局部危机；在竞技场里，它被提升为主要玩法对象。

### 5.2 行为状态机

凶猛鱼具备较细的 AI 状态：

- `roam`：自由游弋
- `detect`：发现玩家
- `stalk`：悄悄接近
- `circle`：环绕观察
- `lunge`：蓄力扑击
- `bite`：咬住玩家
- `devour`：吞食处理
- `retreat`：进食后撤离
- `fear` / `flee`：被灯光驱赶后的受惊与逃离
- `hit`：被打中或被逼退时快速撤离
- `dying`：死亡动画阶段

这套状态机说明敌人设计重点在“行为表演感”，不是简单追踪碰撞。

### 5.3 怕光机制

凶猛鱼会受到手电照射影响，但不是只要进入视野就逃跑，而是结合：

- 与玩家距离
- 在手电扇形范围内的位置
- 手电当前亮度
- 手电是否损坏闪烁

这使得手电既是照明工具，又是对敌手段。

### 5.4 扑击与吞食流程

凶猛鱼攻击有明显的节奏段落：

1. 观察或绕圈
2. 蓄力
3. 冲刺扑击
4. 咬住玩家
5. 吞食与死亡过场

这种设计让玩家有心理预警窗口，也让敌人更像捕猎者，而不是直接贴脸掉血源。

### 5.5 玩家攻击方式

玩家使用氧气瓶挥击进行近战攻击。

特点：

- 扇形判定
- 有攻击角度与范围
- 有冷却
- 有挥击表现与停留残影
- 命中后会触发明显反馈

当前敌鱼设计偏“高机动但脆弱”：常态下很难近身，但一旦被真正打中，就会进入死亡流程。

### 5.6 竞技场中的战斗定位

在竞技场里，这套系统从局部事件转化为主循环：

- 生成敌鱼
- 周旋走位
- 借障碍规避
- 抓扑击间隙反打
- 清完一波后进入下一波

因此，竞技场其实是凶猛鱼 AI 与玩家攻击系统的集中展示场。

## 六、NPC 与同伴设计

### 6.1 NPC 的主要职责

主线中的 NPC 不是纯对话对象，而是承担以下作用：

- 叙事推进
- 目标引导
- 陪伴感与对照感
- 关键救援行为
- 路线示范或误导

### 6.2 熊子

熊子是失踪者，也是驱动主线全部行动的中心角色。

他的设计功能包括：

- 作为被寻找目标提供明确任务动机
- 通过“先一步进入危险区域”制造追逐感
- 通过无法立刻救出形成持续悬念
- 通过可能死亡结局承担悲剧重心

### 6.3 潘子

潘子在第二关中的核心功能是“把危险的不可逆，暂时转回可逆”。

也就是说，他不是一般意义上的陪同 NPC，而是：

- 见证二次下潜
- 参与危险更深层区域
- 在玩家濒死时完成救援
- 让第二关形成结构上的闭环

### 6.4 NPC 的设计原则

NPC 不应喧宾夺主。

他们存在的目的不是持续占据画面中心，而是：

- 在关键时刻推动情绪变化
- 让玩家行为更有叙事对照
- 让“独自下潜”和“有人同行”的差异更明显

## 七、迷宫模式场景辨识度设计稿

这一节用于把 `ToDo.md` 里已经明确提出的“场景辨识度”要求，整理成一份可直接指导程序、美术与 UI 协作的正式设计稿。

这里讨论的重点不是把迷宫做得更花，而是让玩家在一次下潜结束后，能明确说出：

- 我刚才进入的是哪一种空间
- 那个地方为什么危险或值得记住
- 我下次应该去找哪一类线索

### 7.1 设计目标

迷宫模式的场景辨识度，必须同时满足四个目标：

1. **帮助记路**：玩家能靠颜色、材质、空间母题和叙事物件，迅速判断自己大概身处哪一类区域。
2. **服务复盘**：玩家回到岸上后，能把本次探索结果转化成一句能复述的话，而不是只记得“又是蓝黑色的洞”。
3. **保持压迫感**：辨识度不是做成观光洞穴；黑暗、狭窄、泥沙和不确定性仍然是主导情绪。
4. **符合当前工程现实**：当前项目是 `TypeScript + Canvas 2D + 随机迷宫`，所以设计必须能通过程序化元数据和风格化绘制实现，而不是依赖大量逐张手绘资源。

### 7.2 当前项目现状评审

结合当前迷宫模式实现，可以确认下面这些事实：

- 已经存在**稳定的迷宫随机生成**、`mazeMap / mazeWalls / mazeExplored`、跨下潜保留、返岸结算与认知地图展示。
- 迷宫模式已经有**专属渲染分流**，岸上、入水、水下、结算由不同阶段绘制，具备继续分层表现的空间。
- 认知地图已经切换为**手绘风格**，说明“现实真值”和“玩家认知图”已经可以分离表达。
- 当前水下辨识信息仍然偏弱，玩家主要靠绳索和记忆回路判断位置，区域之间的视觉语言还不够稳定。

因此，下一步不需要推翻地图算法或渲染架构，而应采用：

- **迷宫生成后补充场景元数据**
- **渲染层按元数据切换风格**
- **认知地图与返岸摘要同步记录这些风格信息**

这是一条与现有项目状态最匹配、实现风险最低的路线。

### 7.3 总体设计原则

#### 7.3.1 一个区域只给一个主识别句

每个重要区域或洞室，不应同时堆太多视觉关键词，而应保证玩家能快速形成一句主记忆：

- 黄泥双腔洞
- 白石漏斗井
- 红褐事故室
- 神圣空腔

先有**一句话能复述的主识别句**，再谈细节装饰。

#### 7.3.2 静态形态与动态感受必须同时变化

场景辨识度不能只靠换颜色。

至少应由四层信息共同组成：

- **材质底色**：黄泥、白石、红褐沉底、纯白反光
- **空间形态**：双腔、漏斗、狭缝桥接、事故室、空腔
- **动态环境**：颗粒浓度、泥沙厚度、可见度、水流感
- **叙事物件**：脚蹼、破瓶、断绳、手电、遗留工具、生物巢穴痕迹

只有这四层互相咬合，玩家才会觉得“这是另一个地方”，而不是“同一套洞穴换了点颜色”。

#### 7.3.3 先做可重复识别，再做强演出特例

当前优先级应是：

- 先让大量迷宫局都能稳定出现几类**可识别空间单元**
- 再考虑极少数强演出房间

不能一开始就把重点压在少量稀有奇观上，否则大部分游玩时间仍然缺乏辨识度。

#### 7.3.4 岸上与水下的信息职责不同

- **水下**：给玩家即时辨识线索，让其判断位置、风险和是否继续深入
- **岸上**：把这些线索整理成更稳定的认知标签、摘要和地图语言

也就是说，水下负责“感到不一样”，岸上负责“说得清哪里不一样”。

### 7.4 场景辨识度的五层结构

#### 7.4.1 第一层：主岩性层

当前版本不再把“钟乳石、页岩、花岗岩感”混写成同一层主题，而是先定义**主岩性层**。主岩性层负责决定一大片区域的基础轮廓、壁面质感、颗粒气质和认知地图主色。

第一版固定五类主岩性：

1. **黄泥区**
   - 关键词：混浊、温脏、贴壁泥沙
   - 视觉语义：偏黄褐底色、较高颗粒密度、边缘发糊
   - 体验作用：强调方向感差、容易心慌、像刚被扰动过

2. **白石灰岩区**
   - 关键词：冷白、反光、硬、空
   - 视觉语义：浅白岩壁、高反差边缘、冷色反射
   - 体验作用：带来短暂清晰感，但也容易形成“暴露感”

3. **红褐沉积区**
   - 关键词：锈、淤积、事故感、旧痕迹
   - 视觉语义：红褐沉积、低饱和暗红色带、残骸更容易成立
   - 体验作用：适合承载事故室、断绳、旧装备等叙事线索

4. **页岩夹层区**
   - 关键词：薄片、层理、水平切片感
   - 视觉语义：壁面和背景都强调层叠薄片与横向层理
   - 体验作用：让玩家快速识别“这是片状沉积区”，而不是普通暗色岩洞

5. **硬岩块裂区**
   - 关键词：厚重、块裂、冷硬、断面清晰
   - 视觉语义：大块硬岩、棱角轮廓、裂面感强
   - 体验作用：用于承接过去想表达的“花岗岩感”，但语义聚焦在玩家能直接感知的块裂硬岩体验，而不是地质课式命名

设计约束：

- 主岩性层决定的是**区域基本轮廓**，应按整片区域或大洞室分配，而不是一格一换。
- 同一局建议出现 2~3 类主岩性，让玩家能复述差异，但不至于过杂。
- 任何主岩性都不能把画面提亮到破坏洞穴潜水题材的压迫感。

#### 7.4.2 第二层：局部构造层

在主岩性之上，迷宫再叠一层**局部构造层**。这一层不回答“这片区域是什么岩性”，而回答“这段顶壁或局部空间长了什么构造”。

当前优先级最高的局部构造包括：

- **钟乳石带**：成簇垂挂在洞顶或顶壁边缘，强调向下压迫和滴落感
- **石笋堆**：从地面向上形成堆状阻隔，适合作为洞室记忆点
- **流石壁**：顺着墙面形成流淌般的表面组织，强调湿滑和导流感
- **坍塌块堆**：局部堆积的大块坠落岩，强化事故感和绕行感
- **裂缝帘幕**：多条近平行裂隙在远处形成帘状轮廓

关键设计原则：

- **钟乳石不是主岩性。** 它是局部构造，必须表现为洞顶或顶壁边缘的簇状垂挂，而不是把整片区域命名成一种“岩石主题”。
- 局部构造应覆盖在主岩性之上，优先改变轮廓、顶壁压迫和局部记忆点，而不是替代整片区域的基础色温。

#### 7.4.3 第三层：空间母题库

在主岩性和局部构造之上，迷宫再叠一层“空间母题”。当前版本最值得优先做的是这些：

- **双腔洞**：两个小洞袋被窄通道连接，适合作为记忆节点
- **漏斗井**：局部向内收束或向下坠的竖向感区域，强调深入感
- **狭缝桥接洞**：两个区域之间只有窄而长的过渡缝，适合制造心理门槛
- **事故室**：空间略放大，内部带有残留物和不自然痕迹
- **生物巢穴**：墙面或地面有有机聚集感，强调危险和异质感
- **空旷神圣室**：更开阔、更安静，但因为过于空而显得反常

这些母题不要求都变成专门手工关卡，而是应被理解为：

- **随机迷宫中的“命名空间单元”**
- 是程序能识别、渲染能强化、地图能记录的空间类型

#### 7.4.3 第三层：动态环境特征

动态特征用于让玩家不用停下来观察，也能感觉区域变化。

至少应逐步建立以下差异：

- 悬浮颗粒密度差异
- 泥沙颜色和厚度差异
- 光线反射强弱差异
- 局部静水 / 微弱水流感差异
- 可见度变化速度差异

设计要求：

- 动态差异必须服务辨识，不是单纯做特效。
- 第一版优先做**颗粒密度、泥沙颜色、局部明暗反射**三项，避免扩得太散。
- 不同区域的动态参数，应当是“可被玩家感觉到”的，而不是只有代码里数值不同。

#### 7.4.4 第四层：叙事物件簇

叙事物件不应随机均匀撒满地图，而应按“物件簇”组织。

第一版建议优先准备这些可程序绘制的物件：

- 被卡住的脚蹼
- 破损氧气瓶
- 断掉的引导绳
- 掉落手电
- 锤子或简易工具
- 异形水草 / 珊瑚状附着物

设计要求：

- 物件要优先放在**洞室、分叉节点、死路尽头、事故感区域**，不应满地都是。
- 物件首先服务“记忆锚点”，其次才是装饰。
- 第一期可以只做轮廓化、剪影化风格，不要求高细节贴图。

#### 7.4.5 第五层：认知图与摘要同步语言

场景辨识度如果只出现在水下，就会在返岸后立刻损耗。

因此岸上系统必须同步承载：

- 认知地图上的材质区分提示
- 新发现空间母题的命名或标签
- 本次下潜摘要里的“新发现线索”
- 后续可追踪的地标列表或区域标签

如果玩家已经看过“红褐事故室”，返岸后就不该只看到“新探索 28 格”，而应额外看到“发现红褐事故室线索”这类更可复述的信息。

### 7.5 当前版本的可实现方案

结合现在的代码结构，第一版不应尝试“全图每一格都有独特叙事含义”，而应采用**轻量程序化分层**。

#### 7.5.1 第一版推荐只做三类真正会影响体验的内容

1. **区域材质分配**
   - 给迷宫中的大区域或大洞室指定材质谱系
   - 让岩壁、底色、颗粒和局部明暗一起变化

2. **洞室母题标记**
   - 从迷宫结构中识别部分大洞室或关键连接区
   - 为其赋予“双腔洞 / 漏斗井 / 事故室”等标签

3. **叙事物件簇**
   - 在少量关键点生成 1~3 个物件簇
   - 让玩家在复盘时能记住“脚蹼那条路”“破瓶那个洞”

#### 7.5.2 第一版暂时不要做的内容

为了控制复杂度，这一版明确不建议同时做：

- 每局大量独特命名房间
- 复杂可交互道具系统
- 大量稀有演出事件
- 影响碰撞与路线的特殊地形规则
- 独立资源流水线的高成本手绘贴图

这一阶段的重点是**辨识**，不是再开一个大型内容分支。

### 7.6 美术审美约束

#### 7.6.1 不是彩色乐园，而是克制的差异

场景辨识度不等于提高饱和度。

整体美术基调仍应保持：

- 低亮度
- 低至中等饱和度
- 局部高对比而非全屏鲜艳
- 手电与局部反光决定视觉重心

#### 7.6.2 同一类区域要“先看整体，再看细节”

玩家在移动中首先感受到的应该是：

- 整体色温变了
- 墙体质感变了
- 颗粒浓度变了

然后才是：

- 看到了脚蹼
- 看到瓶子
- 看到断绳

所以大形态和大色调优先级高于小装饰。

#### 7.6.3 叙事物件必须可被快速识别成轮廓

由于当前项目是俯视角、Canvas 风格化渲染，因此物件设计必须强调：

- 轮廓清楚
- 俯视识别强
- 颜色不与底色完全融掉
- 一眼能看出“这是人留下来的东西”

### 7.7 对岸上 UI 与认知地图的要求

#### 7.7.1 岸上卡片不再只写数值

当前岸上卡片已有下潜次数、最深到达、绳索和上次摘要。

场景辨识度接入后，岸上卡片或结算摘要应逐步增加：

- 本次新发现区域类型
- 本次新发现空间母题
- 本次新发现叙事物件线索

这样玩家回到岸上时，得到的是“认知增量”，不只是计数增量。

#### 7.7.2 认知地图必须继续坚持“玩家视角”

迷宫认知图不应直接变成无误差的工程图，而应继续保持：

- 已探索与未探索分层
- 手绘风格
- 模糊与确定并存
- 后续可增加区域纹理提示，而不是直接替代掉现有抽象感

地图上的场景辨识度信息，建议用这些方式呈现：

- 局部底纹差异
- 小标签或简名
- 图例补充
- 本次新增高亮

而不是把整张图做成高精确地质剖面图。

### 7.8 程序实现边界与落地顺序

#### 7.8.1 元数据优先，不改核心碰撞规则

当前迷宫模式已经依赖 `mazeMap / mazeWalls / mazeExplored` 完成碰撞、渲染与光照。

场景辨识度第一版最合理的做法是：

- 在迷宫生成完成后，额外生成**区域主题、洞室标签、物件点位**等元数据
- 渲染层读取这些元数据改变表现
- 逻辑层读取这些元数据决定何时记录“已发现线索”

而不是去改动底层碰撞和寻路语义。

#### 7.8.2 先做“生成一次、运行期复用”的数据

为了兼顾性能，所有场景辨识度相关的结构应优先在迷宫生成或重置时完成。

不建议每帧做：

- 大范围区域分析
- 洞室重新识别
- 物件点位重新选取

运行时应以读取和轻量判断为主。

#### 7.8.3 分三步推进

**第一步：主岩性层成立（✅ 已完成）**

- 区域主题分配（基于节点深度分带 + 随机打乱 + BFS 扩散）
- 主岩性固定为 5 类：黄泥区、白石灰岩区、红褐沉积区、页岩夹层区、硬岩块裂区
- 岩壁 / 内壁 / 颗粒 / 认知地图主色全部跟随主岩性变化
- 水下进入新区域时显示提示文字，结算页记录本次新发现主岩性

**第二步：局部构造层成立（✅ 已完成首版）**

- 场景过渡从硬切改为渐变混合，过渡带内按距离比例混色
- 墙体造型差异化继续保留，但语义先由主岩性决定，再由局部构造覆写
- 钟乳石被降级为局部构造，不再当作主岩性主题
- 首版局部构造图已经接入运行态，并在墙体造型中用钟乳石簇强化顶壁压迫感
- 背景装饰层只服务局部气质，不再承担“定义整片区域是什么”的职责

**第三步：空间母题与叙事锚点成立**

- 双腔洞 / 漏斗井 / 事故室等标签
- 大洞室或关键连接空间命名
- 结算摘要出现“新发现空间类型”
- 脚蹼 / 破瓶 / 断绳 / 手电等物件簇
- 岸上认知语言与地图图层联动
- 正式救援前评估开始参考这些线索是否足够

### 7.9 本设计稿的最终判断

从当前项目状态来看，**场景辨识度已经到了必须做、而且适合现在就做的阶段**。

原因不是“想加点美术花活”，而是：

- 多次下潜闭环已经成立
- 玩家开始需要真正记住空间
- 岸上认知图与返岸摘要已经提供了承接位置
- 当前最缺的正是“让一次探索变成一句可复述信息”的中间层

因此，场景辨识度应被视为迷宫模式下一阶段的**核心系统增强**，而不是边角装饰。

## 八、本卷最重要的结论

- **地图不是开放世界，而是服务章节节奏和情绪爬升的引导式洞穴结构。**
- **地标是剧情落地的关键纽带，改地图不能只改形状而不改地标关系。**
- **氧气、手电、狭窄地形、绳索共同构成主线压力系统。**
- **凶猛鱼在主线是危机装置，在竞技场才是主要玩法对象。**
- **迷宫模式下一阶段的重点，不是再加更多事件，而是让不同空间具备可复述、可回忆、可映射到认知图的辨识语言。**
