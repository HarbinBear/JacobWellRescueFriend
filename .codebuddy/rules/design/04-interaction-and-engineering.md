# 设计文档分卷 04：交互、表现与工程约束

## 本卷用途

本卷主要回答下面这些问题：

- 输入与长按为什么这样设计
- UI 与反馈在项目中承担什么作用
- 渲染为什么强调光照和黑暗
- 为什么项目采用集中式状态驱动
- 主循环与模式切换的设计意图是什么
- 后续扩展时哪些原则不能破坏

如果任务与输入、UI、表现、状态流、扩展约束有关，应优先阅读本卷。

---

## 一、输入与交互设计

### 1.1 输入目标

项目面向微信小游戏，因此输入设计必须兼顾：

- 触摸操作
- 虚拟摇杆或触控区域
- 菜单点击
- 长按确认
- 可能存在的键盘调试支持

### 1.2 交互分层

可以把交互理解为三层：

1. **移动控制**：持续输入，影响角色位移与朝向
2. **动作输入**：攻击、确认、长按等瞬时行为
3. **界面输入**：菜单切换、章节选择、模式切换

### 1.3 长按交互的重要性

项目中长按不只是一个输入技巧，而是用于强调“需要决心”的操作，例如：

- 铺设绳索
- 放弃救援

所以后续新增关键选择时，优先考虑是否也该使用长按，而不是轻触。

## 二、UI 与反馈设计

### 2.1 UI 不是单纯信息栏

本项目的 UI 除了展示数值，还承担：

- 引导玩家当前该做什么
- 告诉玩家事件是否可触发
- 表现危险程度
- 承接菜单与章节入口
- 呈现剧情选择与结局态

### 2.2 主要 UI 职责

现有 UI 系统至少覆盖以下内容：

- 主 HUD
- 菜单页面
- 操作说明页
- 绳索长按按钮或进度提示
- 攻击挥击特效
- 结算或失败信息
- 剧情相关提示语

迷宫模式还额外有两个 UI 约定：

- 使用**专属迷宫小地图**，不与旧主线调试小地图叠加显示
- 迷宫小地图默认折叠，只在玩家主动展开时显示完整信息

### 2.3 视觉反馈类型

项目大量依赖即时反馈来补足简化美术条件下的表现力，例如：

- 红屏
- 震动
- 图标提示（问号、感叹号）
- 按钮进度圈
- 闪烁灯光
- 攻击残影

这些反馈是项目体验的重要组成部分，不能轻易删减。

## 三、渲染与氛围设计

### 3.1 渲染目标

Canvas 2D 渲染在这个项目里不是为了复杂 3D 效果，而是为了实现：

- 手工控制的黑暗空间感
- 稳定的俯视角表现
- 可定制的光照遮罩
- 低成本但强氛围的演出

### 3.2 光照是氛围主轴

光照系统至少承担：

- 生成手电可视区域
- 提供遮挡后的不确定性
- 与泥沙、雾感、黑暗叠加
- 为凶猛鱼怕光机制提供玩法依据

在迷宫模式下还要额外满足一个工程约束：

- 光照遮挡必须直接读取迷宫专属地图和迷宫墙体，不能继续沿用主线地图数据，否则就会出现“空中透明岩石挡光”的错位问题

### 3.3 角色与敌人绘制

角色和敌人都采用风格化绘制方式，而非复杂逐帧贴图。

其优点：

- 便于程序驱动状态表现
- 便于依据角度、状态、动画时间做细微变化
- 便于快速调整体型与反馈

### 3.4 绳索与环境层

绳索、墙体、洞穴边缘、光照遮罩共同组成画面的空间可信度。

在这种黑暗潜水题材中，哪怕只是多一条绳索路径，也会显著提升“这个空间有人真实穿行过”的感觉。

迷宫模式对绳索还有一条额外约束：

- 绳索避障和找锚点必须与迷宫地图使用同一套空间数据，否则会出现绕路失效或直接穿岩的问题

## 四、状态管理设计

### 4.1 全局状态树是项目中枢

项目以集中式状态对象驱动。

这意味着：

- 大部分系统通过共享状态通信
- 渲染与逻辑依赖同一批状态数据
- 模式切换实质上是状态切换
- 剧情推进实质上是状态字段变化

### 4.2 需要重点关注的状态域

后续接手时，应优先理解这些状态分区：

- **屏幕或菜单状态**：当前处于主菜单、主线、竞技场、结局还是失败
- **玩家状态**：位置、速度、朝向、生存参数、动作状态
- **剧情状态**：`stage`、剧情文本、演出参数、剧情标记
- **地图状态**：地图数据、墙体、区域、地标、探索信息
- **绳索状态**：当前铺设中绳索、已完成绳索、UI 进度、长按信息
- **敌人状态**：凶猛鱼列表、被咬流程、攻击动画
- **竞技场状态**：波次、计时、击杀、结算相关
- **迷宫状态**：`state.mazeRescue` 下的迷宫地图、迷宫墙体、迷宫探索层、出口、NPC 初始点、小地图展开状态、`shoreMapOpen`（岸上全屏地图是否打开）、`divingInTimer`（入水动效计时帧数）

### 4.3 状态驱动的优点与风险

优点：

- 迭代快
- 事件联动直观
- 适合原型期密集创作

风险：

- 状态字段增多后容易相互耦合
- 新剧情分支可能遗忘清理旧状态
- 多模式共存时容易出现边界污染

因此后续改动时要特别注意重置逻辑与模式切换逻辑。

## 五、主循环与模式切换设计

### 5.1 启动流程

项目入口大致遵循以下流程：

- 初始化纹理
- 注册输入
- 重置游戏逻辑
- 进入持续更新与绘制循环

### 5.2 双更新结构

主循环中至少存在两套逻辑更新语义：

- `update()`：主线相关逻辑
- `updateArena()`：竞技场相关逻辑
- `draw()`：统一渲染出口

这说明该项目不是把竞技场做成完全独立子工程，而是通过同一主循环分流不同模式。

### 5.3 屏幕状态流转

可将 `state.screen` 理解为项目最外层状态机：

```text
menu -> play -> ending / lose
menu -> fishArena -> lose / menu
menu -> mazeRescue -> mazeRescue（结算后重玩）/ menu
```

实际项目中还会叠加章节选择、结局分页、模式返回等细分流程。

迷宫模式还有一个重要约束：

- 进入 `mazeRescue` 后，渲染、光照、绳索、碰撞、小地图都应优先读取 `state.mazeRescue` 内的专属数据，而不是继续复用主线 `state.map`

## 六、工程与资源设计约束

### 6.1 工程依赖保持轻量

当前工程依赖非常轻，核心开发依赖只有 TypeScript。

这说明项目倾向于：

- 原生 Canvas 逻辑
- 少框架、少封装
- 更直接地控制运行时行为

### 6.2 资源目录语义

从目录结构可以推断资源分层大致如下：

- `audio/`：音频资源
- `texture/`：贴图与地图源图
- `scripts/`：构建或素材处理脚本
- `src/`：核心源码
- `typings/`：类型声明
- `.codebuddy/rules/`：供模型理解项目的规则文档

### 6.3 当前阶段的合理认知

这是一个**正在持续扩写内容的单机型创作项目**，不是已经完全工程化的商业大项目。

因此接手时的优先级建议是：

1. 先理解现有行为与设计意图
2. 再补内容或修问题
3. 最后再考虑抽象和重构

### 6.4 版本控制卫生

为了避免把无关的本地文件带进仓库，项目应通过根目录 `.gitignore` 过滤以下内容：

- `node_modules/` 这类本地依赖目录
- `dist/` 与 `*.tsbuildinfo` 这类构建产物或构建缓存
- `.DS_Store`、`Thumbs.db` 这类系统缓存
- `.vscode/`、`.idea/`、`*.swp` 这类编辑器缓存
- `project.private.config.json` 这类微信开发者工具私有配置

这样可以保证仓库主要保留：

- 源码
- 音频与贴图资源
- 项目级配置
- 供后续模型接手的技术与设计文档

## 七、后续扩展建议

### 7.1 扩展剧情时

应优先复用现有结构：

- 继续沿用 `stage`
- 继续沿用地标触发
- 继续沿用 `flags` 记录细节状态
- 尽量让剧情依附在移动、到达、等待、长按、撤退等动作上

### 7.2 扩展地图时

需要优先保证：

- 新区域的情绪作用清晰
- 地标相对关系不被破坏
- 狭窄或宽阔节奏仍可控制
- 光照与撤退路线仍有意义

### 7.3 扩展敌人与战斗时

不应把项目带偏成纯动作游戏。新增战斗内容应继续服务：

- 氛围
- 生存压力
- 深水未知感
- 特定模式下的动作挑战

### 7.4 扩展 UI 时

优先保持：

- 可读性强
- 情绪一致
- 黑暗水下主题一致
- 不要让 HUD 过满破坏沉浸感

## 八、给后续模型的理解优先级

如果是新会话第一次接手这个项目，建议按以下顺序理解：

1. **先理解这是一个“洞穴潜水救援叙事游戏”**，不是普通动作游戏。
2. **再理解 `state.story.stage` 是主线进度主轴**。
3. **再理解地图区域与地标决定剧情发生位置**。
4. **再理解光照、氧气、绳索共同构成探索压力**。
5. **最后再理解竞技场是对敌人与战斗系统的独立玩法化复用**。

如果忽略以上顺序，容易把局部实现误读成项目核心。

## 九、本卷最重要的结论

- **长按、红屏、震动、闪灯这类反馈不是装饰，而是情绪设计的一部分。**
- **项目是集中式状态驱动结构，模式切换本质上是状态切换。**
- **主循环共享一套外层框架，主线与竞技场在其中分流。**
- **后续扩展应优先保持氛围和行为正确，再考虑抽象和重构。**

## 十、本卷补充：多次下潜主线闭环的程序、美术与 UI 拆解稿

这一节用于把“多次侦察、逐步认知、岸上整理、最终救援兑现”的主线闭环，进一步拆成可分工、可排期、可持续扩写的制作稿。

它不是重复前文原则，而是回答三个更直接的问题：

- 程序接下来应该先做哪些系统
- 美术接下来应该提供哪些信息层
- UI 接下来应该如何组织岸上与水下的信息流

### 10.1 先给后续接手者的最短结论

主线不应再被理解为“一次下潜闯通整张洞穴图”，而应理解为：

- 玩家多次进入同一洞穴
- 每次带回一部分确定信息与准备成果
- 岸上把这些成果整理成稳定认知
- 最终把前面所有准备兑现为一次真正的救援撤离

因此，后续无论是地图、美术、绳索、标记、菜单、结算，还是状态管理，都应优先服务**跨下潜认知积累**，而不是只服务单次流程。

### 10.2 三条总制作原则

#### 10.2.1 程序原则：先保证“成果能留下来”

如果玩家探到一半回岸，至少应保留下列成果中的一部分：

- 已探索路径
- 已确认路径
- 已发现洞室与地标
- 已铺绳索
- 已放节点标记
- 危险区记录
- 本次路径轨迹摘要

如果这些成果不能稳定带回岸上，多次下潜就会退化成重复刷图。

#### 10.2.2 美术原则：先保证“空间能被一句话记住”

每个重要空间都不应只是“另一块洞穴背景”，而应具备至少一组可复述的记忆句，例如：

- 黄泥双腔洞
- 白石漏斗井
- 红褐事故室
- 脚蹼死路
- 神圣空腔

如果玩家无法在岸上用一句话说清刚才去过哪里，那说明该区域的辨识信息还不够。

#### 10.2.3 UI 原则：先保证“岸上完整，水下克制”

- 岸上界面负责整理、命名、判断、计划
- 水下界面负责压力、方向、当前动作与局部可依赖信息

不要把岸上该承担的信息处理压力，重新塞回水下 HUD。

### 10.3 程序拆解稿

程序部分建议按“先闭环、再增强”的顺序实现，而不是一开始就追求全部功能完整。

#### 10.3.1 第一批必须完成的系统

##### A. 跨下潜持久化系统

目标：让探路成果能稳定回岸。

至少需要保存：

- 认知图数据
- 已发现地标
- 已确认路径
- 已铺绳索
- 已放节点标记
- 危险区注记
- 各次下潜摘要

设计要求：

- 保存的是**玩家已认知的信息**，不是整张地图真值
- 允许“模糊信息”和“确定信息”并存
- 允许未来扩展玩家手动注释

##### B. 下潜类型系统

当前实现中不再通过独立按钮区分下潜类型，而是统一通过点击洞口触发下潜：

- 系统根据是否已发现NPC自动决定：未发现→`scout`，已发现→`rescue`
- 发现NPC后任何下潜都可以靠近长按绑绳救援
- 后续可考虑进一步细分为 `prepare`（铺路型）、`verify`（验证型）等类型

设计要求：

- 不同类型可影响结算文案和风险容忍度
- `rescue` 的核心区别在于可以绑绳带人，而不是入口按钮不同

##### C. 岸上整理地图界面

目标：让多次下潜形成真正闭环。

至少需要支持：

- 显示已确认路径
- 显示模糊探索轮廓
- 显示绳索与节点标记
- 显示关键地标
- 高亮本次新增发现
- 选择下一次目标

设计要求：

- 系统自动带回本次探索结果
- 玩家在岸上进行整理、确认与命名
- 不建议直接展示“全真自动地图”

##### D. 探路撤离协议

这是你前面提出的“单人一键减压上浮”的正式系统化版本。

建议约束：

- 仅限单人下潜
- 仅在非正式救援状态下可用
- 需要长按触发
- 不能在极窄缝、卡住、异常演出或关键剧情状态中触发
- 触发后直接进入本次下潜结算

设计目的：

- 降低探路挫败
- 鼓励玩家把侦察视为有效行动
- 允许“我先把情报带回去”成为合法策略

##### E. 正式救援状态与结算摘要

正式救援阶段必须补出独立状态，而不是只靠零碎 `flag` 临时拼接。

结算至少应输出：

- 本次下潜类型
- 到达最深处
- 新发现区域数
- 新发现地标数
- 新增绳索与标记数
- 返回原因
- 是否成功带人上浮
- 本次认知图增量

#### 10.3.2 第二批重要增强系统

##### A. 节点标记系统

建议只保留三类核心语义：

- 方向类：指向目标、出口或主路线
- 否定类：表示死路、危险、不要走
- 记忆类：表示关键点、地标、重要判断点

设计要求：

- 标记只能贴在节点，不应允许全地图随意铺满
- 节点至少包括分叉口、洞室入口、狭缝口与特殊地标附近
- 标记应同步记录到岸上认知图

##### B. 洞室命名与简名系统

目标：让玩家在岸上形成稳定的空间语言。

可行方式：

- 系统给出简短候选名
- 玩家选择或微调
- 名称进入地标列表、地图注记与结算摘要

##### C. 路线回放与行动记录系统

目标：让玩家能复盘“我这次到底走了哪条线”。

建议承载内容：

- 本次轨迹
- 折返点
- 最深点
- 标记与绳索铺设点
- 返回原因

##### D. 正式救援前评估状态

建议增加一个明确判断：**是否满足正式救援条件**。

判断可参考：

- 是否找到被困者位置
- 是否存在已确认的回撤路线
- 关键节点是否已具备足够辨识度
- 玩家是否主动选择进入正式救援

#### 10.3.3 第三批表现增强系统

- 手绘地图感增强
- 自动记录与手动整理混合标注
- 岸上摘要动画与轨迹演示
- 区域风格差异强化
- 更清晰的危险区语义显示

### 10.4 程序状态结构建议

如果后续要正式接程序实现，建议围绕下面这组宏观状态设计主线大循环：

- `shore`
- `briefing`
- `dive_scout`
- `dive_prepare`
- `dive_rescue`
- `surface_return`
- `debrief`
- `result`

每次下潜建议记录的运行字段包括：

- `diveType`
- `objective`
- `enteredAt`
- `maxDepthReached`
- `newZonesFound`
- `newLandmarksFound`
- `ropePlacedCount`
- `markersPlacedCount`
- `routeTrace`
- `returnReason`
- `rescuedNpc`
- `mapKnowledgeDelta`

这部分的设计重点不是字段数量，而是要明确区分：

- 本次运行态数据
- 跨下潜永久保留数据
- 正式救援专属状态

### 10.5 美术拆解稿

美术不应再按“整张洞穴统一铺素材”的思路交付，而应按“可记忆空间单元”组织。

#### 10.5.1 第一层：空间母题库

每个重要空间都应有明确母题，至少包括：

- 双腔洞
- 漏斗井
- 狭缝桥接洞
- 空旷神圣室
- 事故室
- 生物巢穴

目标：让关卡和地图在布置空间时，有可以复用的“记忆单位”。

#### 10.5.2 第二层：区域材质库

建议至少稳定区分：

- 黄泥区
- 白石灰岩区
- 红褐沉底区
- 大理石或纯白区

目标：让玩家能通过底色和材质，快速识别自己所处的大区域。

#### 10.5.3 第三层：动态环境特征库

建议逐步补出：

- 悬浮颗粒密度差异
- 泥沙浓度差异
- 水流偏向感
- 反光强弱差异
- 死寂无流区域

目标：让空间不仅在静态上可辨识，也在动态感受上可辨识。

#### 10.5.4 第四层：叙事物件库

建议优先准备：

- 脚蹼
- 破损氧气瓶
- 遗骸
- 断绳
- 手电
- 锤子
- 特征水草或珊瑚

目标：让空间具备叙事钩子，并成为岸上复盘时的记忆锚点。

#### 10.5.5 第五层：信息可视化资源库

建议统一设计：

- 绳索样式
- 节点标记样式
- 岸上手绘地图底板
- 已确认 / 模糊 / 猜测 信息样式
- 新发现高亮样式

目标：保证地图信息、现场标记和结算信息具有一致视觉语言。

### 10.6 UI 拆解稿

UI 的核心任务不是堆按钮，而是帮玩家在不同阶段使用不同密度的信息。

#### 10.6.1 岸上 UI

岸上界面应承担完整的信息处理任务，建议至少包含：

- 已探索认知图
- 本次新增高亮
- 地标与洞室列表
- 绳索与标记总览
- 本次目标选择
- 是否进入正式救援的确认
- 上次下潜摘要

岸上 UI 的关键词是：

- 整理
- 判断
- 命名
- 计划

#### 10.6.2 水下 UI

水下界面应刻意克制，优先只保留：

- 氧气与压力信息
- 当前方向与目标感
- 当前可操作提示
- 局部危险反馈
- 绳索或标记相关上下文提示

不建议在水下长期展示完整地图，否则会削弱洞穴潜水题材最重要的不确定性。

#### 10.6.3 水下交互统一原则

建议逐步收敛为：

- 一个主操作按钮
- 长按触发关键动作
- 用上下文决定当前动作语义
- 必要时通过工具轮盘承载次级功能

这样后续无论是铺绳、放标记、撤离、交互还是特殊剧情操作，都能落在同一套交互语言下。

#### 10.6.4 结算 UI

每次返回岸上后的结算，不应只写“成功 / 失败”，而应强调：

- 这次带回了什么认知成果
- 哪些信息仍不确定
- 下一次最值得验证什么

建议至少呈现：

- 最深点
- 新区域数
- 新地标数
- 新增绳索与标记
- 返程原因
- 下一步建议目标

### 10.7 程序、美术、UI 三线协作接口

为了避免后续三条线各自为政，建议明确这些接口：

#### 10.7.1 程序对美术的需求

- 空间母题枚举
- 区域材质枚举
- 节点标记图形语义
- 叙事物件的最小可识别版本
- 岸上地图图层规范

#### 10.7.2 程序对 UI 的需求

- 岸上认知图图层结构
- 水下操作提示状态
- 结算摘要字段定义
- 正式救援确认流程
- 新发现高亮规则

#### 10.7.3 美术与 UI 共同约束

- 同一地标在场景内、地图上、结算页中的识别语言应尽量一致
- 绳索、标记、危险区的图形语义应固定，不应在不同页面频繁变换
- 岸上与水下可以采用不同信息密度，但不应采用彼此冲突的符号系统

### 10.8 排期建议

如果后续按版本推进，建议优先级如下：

#### 10.8.1 第一阶段：先做闭环成立

- 跨下潜持久化
- 岸上认知图基础页
- 侦察 / 救援 两类下潜分流
- 探路撤离协议
- 返岸结算摘要

#### 10.8.2 第二阶段：先做认知语言成立

- 节点标记系统
- 洞室命名与地标列表
- 场景辨识度强化
- 岸上地图的模糊 / 确定 区分

#### 10.8.3 第三阶段：先做兑现感成立

- 正式救援前评估
- 带人撤离时的压力强化
- 路线回放
- 更强的岸上复盘表现

### 10.9 本节最重要的结论

- **程序第一优先级不是加更多事件，而是先让探路成果可以跨下潜保留。**
- **美术第一优先级不是铺满细节，而是先建立可一句话记住的空间母题。**
- **UI 第一优先级不是增加按钮，而是把岸上整理与水下执行的信息密度明确分开。**
- **多次下潜闭环一旦成立，绳索、地标、标记、结算和正式救援才会真正互相抬价值。**

## 十一、本卷补充：潜水员角色表现与手动挡输入可视化设计稿

这一节用于正式定义 `P2 角色表现` 与 `P1.6 手动挡划水动画联动` 的设计目标，回答两个关键问题：

- 潜水员在俯视角下到底应该长什么样，才能一眼看出是“人在水里”而不是抽象单位
- 用户在手动挡下搓屏输入时，画面应该把这个输入翻译成什么视觉语言，才能既有可读性，又不破坏洞穴潜水题材的沉浸感

本节的核心判断是：

> **手动挡输入的主可视化，不应是把手势路径高亮画在屏幕上，而应是把输入翻译成潜水员真实的划水动作、姿态修正和局部水流扰动。**

也就是说，玩家看到的重点应是“这个人正在用身体推水”，而不是“系统正在展示一个输入向量”。

### 11.1 设计目标

潜水员角色表现需要同时满足四个目标：

1. **身份可读**：玩家一眼能认出这是潜水员，而不是普通人形或抽象图标。
2. **朝向可读**：玩家能立刻判断头朝哪、身体朝哪、当前是在推进、滑行还是掉头。
3. **动作可读**：玩家能看出划水、漂浮、修正方向、掉头发力这些不同状态。
4. **输入可读**：玩家手动挡每次搓屏，都能在角色动作上得到即时回应，而不是只在速度变化上感知输入生效。

### 11.2 总体原则

#### 11.2.1 输入要“身体化”，不要“界面化”

手动挡的主反馈必须来自角色身体：

- 手臂推水
- 蛙鞋蹬水
- 身体收紧与释放
- 局部水流被拨开的短暂痕迹

不应把主要表现做成：

- 常驻箭头
- 明显输入轨迹线
- 大面积方向扇形
- 长时间停留的手势提示

这些强 UI 提示会把体验从“我在洞穴里划水”拉回“我在屏幕上发移动指令”，削弱题材沉浸感。

#### 11.2.2 角色不是持续循环游泳，而是“划一下，滑一段，再修正”

当前手动挡的体验目标不是传统摇杆式持续游泳，而是：

- 玩家搓一下
- 角色划一下
- 获得一段推力
- 进入短暂滑行
- 再根据下一次输入继续推进或修正姿态

因此角色动画不应只是无脑循环，而要有明显的“发力相”和“滑行相”。

#### 11.2.3 反向输入不表现为倒车，而表现为刹车与掉头

当玩家朝身体后方反向搓屏时，视觉上不应像角色在原地倒着游。

更合理的表现是：

- 先减速
- 身体扭转
- 单侧或双侧更大幅度发力
- 再逐步完成掉头

这样才能符合洞穴潜水的写实感，也更符合“身体朝向被动跟随速度方向”的整体设计思路。

### 11.3 潜水员俯视角美术结构

#### 11.3.1 轮廓结构

潜水员整体轮廓应采用**前圆、中厚、后分叉**的俯视结构。

可概括为：

- **前端**：头部与面镜，负责朝向识别
- **中段**：肩背、躯干与气瓶，负责身份识别
- **后端**：双腿与双蛙鞋，负责推进节奏识别

轮廓目标不是解剖写实，而是**小尺寸下的可读性优先**。

#### 11.3.2 比例原则

建议角色整体长度以当前版本为基础，放大到约 `1.25 ~ 1.45` 倍视觉长度，但不要夸张到像大体型单位。

推荐比例：

- **头部与面镜区**：约 `18%`
- **躯干与气瓶区**：约 `34%`
- **腰胯过渡区**：约 `12%`
- **腿部与蛙鞋区**：约 `36%`

这个比例的目的不是写实人体，而是确保：

- 头部方向明确
- 中段身份明确
- 后段动作明确

#### 11.3.3 头部与面镜

头部必须承担最强的朝向识别职责。

建议：

- 头盔轮廓偏深色，作为稳定外形
- 面镜以偏前探的椭圆玻璃表示，不放在脸部正中
- 面镜带一个很克制的高光点，作为前方识别锚点

这样即使角色在轻微摆动，玩家也能迅速判断“头朝哪边”。

#### 11.3.4 躯干与肩背

躯干应采用流线水滴形，而不是矩形方块。

目标是表现：

- 人在水中是贴近水流、收紧身体的
- 肩背略宽、腰部收窄
- 身体不是站立人物，而是俯卧游动状态

色彩应保持低亮度、低饱和度，不让躯干本身抢过面镜和手电的视觉重心。

#### 11.3.5 气瓶

气瓶是潜水员身份识别的关键要素。

第一版建议仍以一个主瓶为主体，但造型应从“贴上的矩形块”升级为“背部结构的一部分”：

- 瓶体是圆角长罐
- 顶部有简化瓶阀或金属帽
- 与肩背之间有绑带感
- 位置略偏后，不压到头部区域

如果需要增强识别，也可保留主副瓶的视觉差异，但重点仍是整体一眼可读，不追求装备细节堆砌。

#### 11.3.6 手臂

手臂是手动挡输入最核心的视觉转译部位。

原则：

- 上臂短而略粗
- 前臂更细
- 手掌可简化为短梭形末端
- 默认姿态微张开、微屈肘，像在水里维持平衡

手臂不需要解剖级真实，但必须在小尺寸下能表现：

- 发力
- 回收
- 单侧修正
- 掉头时的更大摆幅

#### 11.3.7 腿部与蛙鞋

蛙鞋负责提供最强的后端节奏感。

建议：

- 腿偏短
- 蛙鞋偏长、偏宽
- 双蹼后端要有分叉感，不并成一个整体
- 形状适合做开合、回收和蹬水的相位变化

在小尺寸俯视角下，蛙鞋是“角色正在动”的关键读数之一。

### 11.4 动作语义设计

潜水员动画不应只有一个持续循环，而应拆成明确的语义状态。

#### 11.4.1 漂浮待机态

当玩家没有明显输入，或速度已经很低时，角色应进入漂浮待机态：

- 手臂轻微外摆或回收
- 蛙鞋有很小的浮动
- 身体有非常弱的左右摆和上下轻漂
- 整体像在维持水中平衡，而不是主动游动

关键词是：**漂、稳、轻微不安定**。

#### 11.4.2 划水推进态

当玩家产生有效搓屏输入时，应触发一次明确的划水推进态：

- 某一侧手臂或双臂出现推水相位
- 双蹼出现短促发力
- 躯干有轻微收紧与释放
- 身后或肢体后方出现短暂扰流水痕

关键词是：**发力、推水、推进**。

#### 11.4.3 滑行态

一次有效划水结束后，如果角色仍有速度但没有新输入，应进入滑行态：

- 四肢回收
- 身体拉长
- 动作明显比推进态平静
- 仍保留很弱的惯性摆动

关键词是：**滑、顺、水阻缓慢接管**。

这能让手动挡表现成“划一下，滑出去”，而不是“有输入才像活着，没输入就像木块”。

#### 11.4.4 掉头修正态

当输入与当前身体朝向夹角较大时，应进入掉头修正态：

- 身体先扭，再逐步顺过去
- 单侧手臂动作更大
- 双蹼开合更明显
- 头部和速度方向允许短时不完全一致

关键词是：**拧身、修正、对齐**。

### 11.5 手动挡输入可视化方案

手动挡输入可视化建议分成三层：

#### 11.5.1 第一层：角色身体反馈（主反馈）

这是最重要的一层，也是默认必须可见的一层。

当玩家搓屏时，画面主要反馈应为：

- 手臂推水动作被触发
- 蛙鞋对应发力
- 身体姿态轻微收紧
- 朝向修正开始发生

玩家应该从角色本体就看出：

> “我刚刚这一下输入，潜水员真的划了一下。”

#### 11.5.2 第二层：局部水流反馈（辅反馈）

为了增强输入与推进的因果关系，可增加短时局部反馈：

- 手臂或蛙鞋后方极短的水纹
- 一小串很淡的气泡
- 局部泥沙被轻微搅动
- 极短命的推进扰流

约束：

- 生命周期短
- 范围小
- 不发光、不夸张
- 更像水被拨开，而不是喷气尾焰

#### 11.5.3 第三层：弱输入提示（教学或适应阶段）

这一层不应是常驻主表现，而应作为可选辅助层，仅用于上手期或必要提示：

- 手指附近极淡的弧线提示
- 一瞬的反向水流指示
- 角色周围非常弱的扇形扰动

这层的目标只是帮助玩家理解“搓动会如何转译成身体动作”，不应喧宾夺主。

### 11.6 输入与动作的映射关系

#### 11.6.1 同向输入

当输入大体与当前身体朝向一致时：

- 双臂轻度同步划水
- 双蹼做稳定发力
- 身体只做轻微压入和释放
- 视觉重点是顺畅推进

这表示角色正在顺着已有方向继续前进。

#### 11.6.2 侧向输入

当输入与身体朝向有明显夹角时：

- 靠近发力方向的一侧手臂动作更大
- 身体出现轻度扭转
- 蛙鞋不做大开大合，而更像辅助修正
- 视觉重点是调整航向，而不是爆发推进

这表示角色在水中“侧修”。

#### 11.6.3 反向输入

当输入接近角色后方时：

- 动作不应表现成倒着后退
- 应表现成先刹车、再拧身、再重新对齐
- 手臂摆幅更大
- 蛙鞋回收和再蹬更明显

这表示角色在水下艰难掉头，而不是游戏图标简单反向移动。

#### 11.6.4 双指交替输入

双指输入不应只被看作“更强的同类输入”，而应尽量转译成更接近真实的双侧交替划水：

- 左右臂或左右下肢可轮流承担主发力
- 身体允许出现很轻微的左右滚摆
- 推进节奏更连续，而不是瞬时跳跃更快

玩家看到的应是“这个人正在连续、交替地划”，而不是“系统数值乘了二”。

#### 11.6.5 本轮细化结论：单次输入、左右腿精确匹配、前进与拐弯线性叠加

在第一版实现基础上，本轮再明确以下约束，并以后者为准：

- **头部不保留额外高亮小点**。面镜本身承担朝向识别，不再在头部额外放一个灰点或亮点。
- **手部不保留肉色小点**。手臂只作为整体轮廓的一部分，不再用高对比末端色块强调手掌。
- **手臂不承担主推进动作**。手只做配合腿部的轻微摆动与转向修正，不再表现成明显划水。
- **主推进动作来自腿部的上下踢水**。不是左右扫腿，而是更接近专业潜水中的上下打水与蹬水。
- **一次输入只对应一次完整踢水**。输入即使不松手，也不能重复驱动第二次踢水；必须等新的输入生命周期开始，才能触发下一次。
- **左右腿必须精确匹配触点节奏**。输入发生在角色局部坐标的左侧，就由左腿承担主踢水；输入发生在角色局部坐标的右侧，就由右腿承担主踢水；双侧同时输入，就允许双腿同时发力。这里的左右是相对角色身体，而不是相对手机屏幕。关键不是预设几种模式，而是程序层必须逐个触点跟踪和消费输入，让任意左右节奏都能被正确映射。
- **输入进度要与踢水进度同步**。踢水不是独立播放的固定时长动画，而要跟随本次输入的累计有效行程推进；输入到哪，踢水就做到哪。整段有效行程都应持续生效，不能只让前半段有移动和表现、后半段突然失效。
- **有效行程必须可调**。一次输入对应一次踢水生命周期，但这个生命周期覆盖多长的输入距离，需要提供明确参数入口，便于根据手感把行程拉长或缩短。
- **同一段输入距离下，输入速度仍要影响结果**。同样长的有效行程，搓得更快时应产生更强的推进与转向；搓得更慢时仍然生效，但力度更柔和。
- **前进与拐弯要拆成两种可叠加动作语义**。
  - 输入向量中与身体朝向同向的前向分量，转译为推进踢水。
  - 输入向量中的侧向分量，转译为手脚一起参与的转向修正动作。
  - 两者不是二选一，而是线性叠加：一次输入可以同时含有前进和拐弯成分。
- **后向输入暂不设计倒退动作**。凡是后向分量，统一在移动与表现上折算进转向修正，而不是表现成角色倒着游。

### 11.7 审美与表现约束

#### 11.7.1 整体低亮度、局部高识别

角色整体仍属于洞穴潜水题材，应保持：

- 低亮度
- 低到中等饱和度
- 不使用大面积高纯度颜色

真正承担识别的是：

- 面镜
- 气瓶
- 蛙鞋末端
- 局部高光

#### 11.7.2 不要做成魔法推进

所有推进反馈都应强调“水被推开”，不要做成：

- 强发光尾焰
- 过长拖尾
- 明显能量条纹
- 夸张喷射流

角色是在黑暗水域里费力划水，不是在科幻空间里点火冲刺。

#### 11.7.3 输入辅助不能长期盖过角色本体

如果后续保留输入提示层，必须遵守：

- 默认弱显示
- 只在输入发生的瞬间短暂出现
- 强度低于角色动作反馈和环境反馈

任何时候都不应让玩家更多地在看“操作提示层”，而不是在看潜水员本身。

### 11.8 第一版实现优先级建议

#### 11.8.1 先完成轮廓重建

第一优先级是重建潜水员可读轮廓：

- 头部 / 面镜
- 躯干 / 肩背
- 气瓶
- 双臂
- 双腿 / 双蛙鞋

先解决“这是不是潜水员、朝哪边”的问题，再谈细动画。

#### 11.8.2 再完成动作语义

第二优先级是补足四种主要动作语义：

- 漂浮待机
- 划水推进
- 滑行
- 掉头修正

先保证玩家能从轮廓和节奏读懂状态，再继续打磨细部流体反馈。

#### 11.8.3 最后接手动挡输入联动

第三优先级是把输入正式挂到角色动作上：

- 有效搓屏触发划水脉冲
- 单指与双指输入影响动作节奏
- 反向搓屏触发刹车与掉头表现
- 局部水流反馈与弱输入提示作为附加层接入

### 11.9 本节最重要的结论

- **手动挡输入的主可视化，应当是潜水员的划水动作与姿态修正，而不是强 UI 化的手势轨迹。**
- **角色轮廓必须优先强化头部面镜、背部气瓶和双蛙鞋三组识别锚点。**
- **动画应拆成漂浮待机、划水推进、滑行、掉头修正四种语义，而不是只做一个持续游泳循环。**
- **反向输入的视觉含义是刹车与掉头，不是简单倒车。**
- **所有辅助手势提示都应弱于角色本体与局部水流反馈，避免破坏洞穴潜水题材的沉浸感。**

### 11.10 第一版细调补充约束（惯性、限速与传导感）

在第一版已经接通手动挡与潜水员表现链路之后，后续细调必须继续遵守下面这些补充约束：

- **移动应更依赖单次输入后的惯性。** 玩家不应为了维持巡航速度而持续高频搓屏；一次有效输入完成后，角色应保留更明显的前冲余速，让玩家可以更从容地给节奏。
- **踢水动作本身必须有速度上限。** 即使输入速度很快，腿部和脚蹼动作也不能无限跟着变快，否则会失去潜水动作的重量感，退化成抽搐式快踢。
- **输入结束后要保留慢后摇。** 单次输入的有效行程结束后，动作不应立刻归零，而应通过较慢的回收和衰减继续承载剩余惯性，让视觉节奏和移动余速保持一致。
- **手臂应随速度变化自动收拢。** 速度越高，手臂越贴近身体，突出流线感；速度降下来以后，手臂才重新张开，承担低速稳定姿态。
- **腿部推进不应再理解为抬腿放腿。** 正确的重点是身体扭动带动大腿、小腿和脚蹼形成前后传导的发力链，尤其要突出脚蹼前收与后扫的远近变化，以及力量从身体传到末端的感觉。
- **左右腿分配应按输入顺序交替。** 每次新输入开始时，都应把发力主腿切换到另一侧；上一次输入如果驱动左腿，这一次新输入就驱动右腿，不再依赖屏幕左右或角色局部左右判定。
- **头部不能再出现额外灰点锚点。** 头部只保留必要轮廓与面镜识别，不再额外绘制会在小尺寸下抢眼的小点或高光。
