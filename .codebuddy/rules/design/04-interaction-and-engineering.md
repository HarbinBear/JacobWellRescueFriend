- 输入与长按为什么这样设计

## 一、输入与交互设计

### 1.1 输入目标

项目面向微信小游戏，因此输入设计必须兼顾：

- 触摸操作
- 虚拟摇杆或触控区域
- 菜单点击
- 长按确认
- 可能存在的键盘调试支持

### 1.2 交互分层

可以把交互理解为三层：

1. **移动控制**：持续输入，影响角色位移与朝向
2. **动作输入**：攻击、确认、长按等瞬时行为
3. **界面输入**：菜单切换、章节选择、模式切换

### 1.3 长按交互的重要性

项目中长按不只是一个输入技巧，而是用于强调“需要决心”的操作，例如：

- 铺设绳索
- 放弃救援

所以后续新增关键选择时，优先考虑是否也该使用长按，而不是轻触。

## 二、UI 与反馈设计

### 2.1 UI 不是单纯信息栏

本项目的 UI 除了展示数值，还承担：

- 引导玩家当前该做什么
- 告诉玩家事件是否可触发
- 表现危险程度
- 承接菜单与章节入口
- 呈现剧情选择与结局态

### 2.2 主要 UI 职责

现有 UI 系统至少覆盖以下内容：

- 主 HUD
- 菜单页面
- 操作说明页
- 绳索长按按钮或进度提示
- 攻击挥击特效
- 结算或失败信息
- 剧情相关提示语

迷宫模式还额外有两个 UI 约定：

- 使用**专属迷宫小地图**，不与旧主线调试小地图叠加显示
- 迷宫小地图默认折叠，只在玩家主动展开时显示完整信息

### 2.3 视觉反馈类型

项目大量依赖即时反馈来补足简化美术条件下的表现力，例如：

- 红屏
- 震动
- 图标提示（问号、感叹号）
- 按钮进度圈
- 闪烁灯光
- 攻击残影

这些反馈是项目体验的重要组成部分，不能轻易删减。

## 三、渲染与氛围设计

### 3.1 渲染目标

Canvas 2D 渲染在这个项目里不是为了复杂 3D 效果，而是为了实现：

- 手工控制的黑暗空间感
- 稳定的俯视角表现
- 可定制的光照遮罩
- 低成本但强氛围的演出

### 3.2 光照是氛围主轴

光照系统至少承担：

- 生成手电可视区域
- 提供遮挡后的不确定性
- 与泥沙、雾感、黑暗叠加
- 为凶猛鱼怕光机制提供玩法依据

在迷宫模式下还要额外满足一个工程约束：

- 光照遮挡必须直接读取迷宫专属地图和迷宫墙体，不能继续沿用主线地图数据，否则就会出现“空中透明岩石挡光”的错位问题

### 3.3 角色与敌人绘制

角色和敌人都采用风格化绘制方式，而非复杂逐帧贴图。

其优点：

- 便于程序驱动状态表现
- 便于依据角度、状态、动画时间做细微变化
- 便于快速调整体型与反馈

### 3.4 绳索与环境层

绳索、墙体、洞穴边缘、光照遮罩共同组成画面的空间可信度。

在这种黑暗潜水题材中，哪怕只是多一条绳索路径，也会显著提升“这个空间有人真实穿行过”的感觉。

迷宫模式对绳索还有一条额外约束：

- 绳索避障和找锚点必须与迷宫地图使用同一套空间数据，否则会出现绕路失效或直接穿岩的问题

## 四、状态管理设计

### 4.1 全局状态树是项目中枢

项目以集中式状态对象驱动。

这意味着：

- 大部分系统通过共享状态通信
- 渲染与逻辑依赖同一批状态数据
- 模式切换实质上是状态切换
- 剧情推进实质上是状态字段变化

### 4.2 需要重点关注的状态域

后续接手时，应优先理解这些状态分区：

- **屏幕或菜单状态**：当前处于主菜单、主线、竞技场、结局还是失败
- **玩家状态**：位置、速度、朝向、生存参数、动作状态
- **剧情状态**：`stage`、剧情文本、演出参数、剧情标记
- **地图状态**：地图数据、墙体、区域、地标、探索信息
- **绳索状态**：当前铺设中绳索、已完成绳索、UI 进度、长按信息
- **敌人状态**：凶猛鱼列表、被咬流程、攻击动画
- **竞技场状态**：波次、计时、击杀、结算相关
- **迷宫状态**：`state.mazeRescue` 下的迷宫地图、迷宫墙体、迷宫探索层、出口、NPC 初始点、小地图展开状态、`shoreMapOpen`（岸上全屏地图是否打开）、`divingInTimer`（入水动效计时帧数）

### 4.3 状态驱动的优点与风险

优点：

- 迭代快
- 事件联动直观
- 适合原型期密集创作

风险：

- 状态字段增多后容易相互耦合
- 新剧情分支可能遗忘清理旧状态
- 多模式共存时容易出现边界污染

因此后续改动时要特别注意重置逻辑与模式切换逻辑。

## 五、主循环与模式切换设计

### 5.1 启动流程

项目入口大致遵循以下流程：

- 初始化纹理
- 注册输入
- 重置游戏逻辑
- 进入持续更新与绘制循环

### 5.2 双更新结构

主循环中至少存在两套逻辑更新语义：

- `update()`：主线相关逻辑
- `updateArena()`：竞技场相关逻辑
- `draw()`：统一渲染出口

这说明该项目不是把竞技场做成完全独立子工程，而是通过同一主循环分流不同模式。

### 5.3 屏幕状态流转

可将 `state.screen` 理解为项目最外层状态机：

```text
menu -> play -> ending / lose
menu -> fishArena -> lose / menu
menu -> mazeRescue -> mazeRescue（结算后重玩）/ menu
```

实际项目中还会叠加章节选择、结局分页、模式返回等细分流程。

迷宫模式还有一个重要约束：

- 进入 `mazeRescue` 后，渲染、光照、绳索、碰撞、小地图都应优先读取 `state.mazeRescue` 内的专属数据，而不是继续复用主线 `state.map`

## 六、工程与资源设计约束

### 6.1 工程依赖保持轻量

当前工程依赖非常轻，核心开发依赖只有 TypeScript。

这说明项目倾向于：

- 原生 Canvas 逻辑
- 少框架、少封装
- 更直接地控制运行时行为

### 6.2 资源目录语义

从目录结构可以推断资源分层大致如下：

- `audio/`：音频资源
- `texture/`：贴图与地图源图
- `scripts/`：构建或素材处理脚本
- `src/`：核心源码
- `typings/`：类型声明
- `.codebuddy/rules/`：供模型理解项目的规则文档

### 6.3 当前阶段的合理认知

这是一个**正在持续扩写内容的单机型创作项目**，不是已经完全工程化的商业大项目。

因此接手时的优先级建议是：

1. 先理解现有行为与设计意图
2. 再补内容或修问题
3. 最后再考虑抽象和重构

### 6.4 版本控制卫生

为了避免把无关的本地文件带进仓库，项目应通过根目录 `.gitignore` 过滤以下内容：

- `node_modules/` 这类本地依赖目录
- `dist/` 与 `*.tsbuildinfo` 这类构建产物或构建缓存
- `.DS_Store`、`Thumbs.db` 这类系统缓存
- `.vscode/`、`.idea/`、`*.swp` 这类编辑器缓存
- `project.private.config.json` 这类微信开发者工具私有配置

这样可以保证仓库主要保留：

- 源码
- 音频与贴图资源
- 项目级配置
- 供后续模型接手的技术与设计文档

## 七、后续扩展建议

### 7.1 扩展剧情时

应优先复用现有结构：

- 继续沿用 `stage`
- 继续沿用地标触发
- 继续沿用 `flags` 记录细节状态
- 尽量让剧情依附在移动、到达、等待、长按、撤退等动作上

### 7.2 扩展地图时

需要优先保证：

- 新区域的情绪作用清晰
- 地标相对关系不被破坏
- 狭窄或宽阔节奏仍可控制
- 光照与撤退路线仍有意义

### 7.3 扩展敌人与战斗时

不应把项目带偏成纯动作游戏。新增战斗内容应继续服务：

- 氛围
- 生存压力
- 深水未知感
- 特定模式下的动作挑战

### 7.4 扩展 UI 时

优先保持：

- 可读性强
- 情绪一致
- 黑暗水下主题一致
- 不要让 HUD 过满破坏沉浸感

## 八、给后续模型的理解优先级

如果是新会话第一次接手这个项目，建议按以下顺序理解：

1. **先理解这是一个“洞穴潜水救援叙事游戏”**，不是普通动作游戏。
2. **再理解 `state.story.stage` 是主线进度主轴**。
3. **再理解地图区域与地标决定剧情发生位置**。
4. **再理解光照、氧气、绳索共同构成探索压力**。
5. **最后再理解竞技场是对敌人与战斗系统的独立玩法化复用**。

如果忽略以上顺序，容易把局部实现误读成项目核心。
