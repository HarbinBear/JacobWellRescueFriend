# 设计文档分卷 05：多次下潜闭环的程序、美术与 UI 拆解稿

## 本卷用途

本卷是从 `04-interaction-and-engineering.md` 拆出的专项制作稿，主要回答下面这些问题：

- 多次下潜闭环的核心制作原则是什么
- 程序应该先做哪些系统，按什么顺序
- 美术应该按什么层次组织空间辨识信息
- UI 应该如何分配岸上与水下的信息密度
- 三条线如何协作，接口在哪里

如果任务涉及多次下潜闭环、跨下潜持久化、岸上整理地图、正式救援状态、程序/美术/UI 协作，应优先阅读本卷。

---

## 一、先给后续接手者的最短结论

主线不应再被理解为"一次下潜闯通整张洞穴图"，而应理解为：

- 玩家多次进入同一洞穴
- 每次带回一部分确定信息与准备成果
- 岸上把这些成果整理成稳定认知
- 最终把前面所有准备兑现为一次真正的救援撤离

因此，后续无论是地图、美术、绳索、标记、菜单、结算，还是状态管理，都应优先服务**跨下潜认知积累**，而不是只服务单次流程。

---

## 二、三条总制作原则

### 2.1 程序原则：先保证"成果能留下来"

如果玩家探到一半回岸，至少应保留下列成果中的一部分：

- 已探索路径
- 已确认路径
- 已发现洞室与地标
- 已铺绳索
- 已放节点标记
- 危险区记录
- 本次路径轨迹摘要

如果这些成果不能稳定带回岸上，多次下潜就会退化成重复刷图。

### 2.2 美术原则：先保证"空间能被一句话记住"

每个重要空间都不应只是"另一块洞穴背景"，而应具备至少一组可复述的记忆句，例如：

- 黄泥双腔洞
- 白石漏斗井
- 红褐事故室
- 脚蹼死路
- 神圣空腔

如果玩家无法在岸上用一句话说清刚才去过哪里，那说明该区域的辨识信息还不够。

### 2.3 UI 原则：先保证"岸上完整，水下克制"

- 岸上界面负责整理、命名、判断、计划
- 水下界面负责压力、方向、当前动作与局部可依赖信息

不要把岸上该承担的信息处理压力，重新塞回水下 HUD。

## 三、程序拆解稿

程序部分建议按"先闭环、再增强"的顺序实现，而不是一开始就追求全部功能完整。

### 3.1 第一批必须完成的系统

#### A. 跨下潜持久化系统

目标：让探路成果能稳定回岸。

至少需要保存：

- 认知图数据
- 已发现地标
- 已确认路径
- 已铺绳索
- 已放节点标记
- 危险区注记
- 各次下潜摘要

设计要求：

- 保存的是**玩家已认知的信息**，不是整张地图真值
- 允许"模糊信息"和"确定信息"并存
- 允许未来扩展玩家手动注释

#### B. 下潜类型系统

当前实现中不再通过独立按钮区分下潜类型，而是统一通过点击洞口触发下潜：

- 系统根据是否已发现NPC自动决定：未发现→`scout`，已发现→`rescue`
- 发现NPC后任何下潜都可以靠近长按绑绳救援
- 后续可考虑进一步细分为 `prepare`（铺路型）、`verify`（验证型）等类型

设计要求：

- 不同类型可影响结算文案和风险容忍度
- `rescue` 的核心区别在于可以绑绳带人，而不是入口按钮不同

#### C. 岸上整理地图界面

目标：让多次下潜形成真正闭环。

至少需要支持：

- 显示已确认路径
- 显示模糊探索轮廓
- 显示绳索与节点标记
- 显示关键地标
- 高亮本次新增发现
- 选择下一次目标

设计要求：

- 系统自动带回本次探索结果
- 玩家在岸上进行整理、确认与命名
- 不建议直接展示"全真自动地图"

#### D. 探路撤离协议

这是"单人一键减压上浮"的正式系统化版本。

建议约束：

- 仅限单人下潜
- 仅在非正式救援状态下可用
- 需要长按触发
- 不能在极窄缝、卡住、异常演出或关键剧情状态中触发
- 触发后直接进入本次下潜结算

设计目的：

- 降低探路挫败
- 鼓励玩家把侦察视为有效行动
- 允许"我先把情报带回去"成为合法策略

#### E. 正式救援状态与结算摘要

正式救援阶段必须补出独立状态，而不是只靠零碎 `flag` 临时拼接。

结算至少应输出：

- 本次下潜类型
- 到达最深处
- 新发现区域数
- 新发现地标数
- 新增绳索与标记数
- 返回原因
- 是否成功带人上浮
- 本次认知图增量

### 3.2 第二批重要增强系统

#### A. 节点标记系统

建议只保留三类核心语义：

- 方向类：指向目标、出口或主路线
- 否定类：表示死路、危险、不要走
- 记忆类：表示关键点、地标、重要判断点

设计要求：

- 标记只能贴在节点，不应允许全地图随意铺满
- 节点至少包括分叉口、洞室入口、狭缝口与特殊地标附近
- 标记应同步记录到岸上认知图

#### B. 洞室命名与简名系统

目标：让玩家在岸上形成稳定的空间语言。

可行方式：

- 系统给出简短候选名
- 玩家选择或微调
- 名称进入地标列表、地图注记与结算摘要

#### C. 路线回放与行动记录系统

目标：让玩家能复盘"我这次到底走了哪条线"。

建议承载内容：

- 本次轨迹
- 折返点
- 最深点
- 标记与绳索铺设点
- 返回原因

#### D. 正式救援前评估状态

建议增加一个明确判断：**是否满足正式救援条件**。

判断可参考：

- 是否找到被困者位置
- 是否存在已确认的回撤路线
- 关键节点是否已具备足够辨识度
- 玩家是否主动选择进入正式救援

### 3.3 第三批表现增强系统

- 手绘地图感增强
- 自动记录与手动整理混合标注
- 岸上摘要动画与轨迹演示
- 区域风格差异强化
- 更清晰的危险区语义显示

### 3.4 程序状态结构建议

如果后续要正式接程序实现，建议围绕下面这组宏观状态设计主线大循环：

- `shore`
- `briefing`
- `dive_scout`
- `dive_prepare`
- `dive_rescue`
- `surface_return`
- `debrief`
- `result`

每次下潜建议记录的运行字段包括：

- `diveType`
- `objective`
- `enteredAt`
- `maxDepthReached`
- `newZonesFound`
- `newLandmarksFound`
- `ropePlacedCount`
- `markersPlacedCount`
- `routeTrace`
- `returnReason`
- `rescuedNpc`
- `mapKnowledgeDelta`

这部分的设计重点不是字段数量，而是要明确区分：

- 本次运行态数据
- 跨下潜永久保留数据
- 正式救援专属状态

## 四、美术拆解稿

美术不应再按"整张洞穴统一铺素材"的思路交付，而应按"可记忆空间单元"组织。

### 4.1 第一层：空间母题库

每个重要空间都应有明确母题，至少包括：

- 双腔洞
- 漏斗井
- 狭缝桥接洞
- 空旷神圣室
- 事故室
- 生物巢穴

目标：让关卡和地图在布置空间时，有可以复用的"记忆单位"。

### 4.2 第二层：区域材质库

建议至少稳定区分：

- 黄泥区
- 白石灰岩区
- 红褐沉底区
- 大理石或纯白区

目标：让玩家能通过底色和材质，快速识别自己所处的大区域。

### 4.3 第三层：动态环境特征库

建议逐步补出：

- 悬浮颗粒密度差异
- 泥沙浓度差异
- 水流偏向感
- 反光强弱差异
- 死寂无流区域

目标：让空间不仅在静态上可辨识，也在动态感受上可辨识。

### 4.4 第四层：叙事物件库

建议优先准备：

- 脚蹼
- 破损氧气瓶
- 遗骸
- 断绳
- 手电
- 锤子
- 特征水草或珊瑚

目标：让空间具备叙事钩子，并成为岸上复盘时的记忆锚点。

### 4.5 第五层：信息可视化资源库

建议统一设计：

- 绳索样式
- 节点标记样式
- 岸上手绘地图底板
- 已确认 / 模糊 / 猜测 信息样式
- 新发现高亮样式

目标：保证地图信息、现场标记和结算信息具有一致视觉语言。

## 五、UI 拆解稿

UI 的核心任务不是堆按钮，而是帮玩家在不同阶段使用不同密度的信息。

### 5.1 岸上 UI

岸上界面应承担完整的信息处理任务，建议至少包含：

- 已探索认知图
- 本次新增高亮
- 地标与洞室列表
- 绳索与标记总览
- 本次目标选择
- 是否进入正式救援的确认
- 上次下潜摘要

岸上 UI 的关键词是：**整理、判断、命名、计划**。

### 5.2 水下 UI

水下界面应刻意克制，优先只保留：

- 氧气与压力信息
- 当前方向与目标感
- 当前可操作提示
- 局部危险反馈
- 绳索或标记相关上下文提示

不建议在水下长期展示完整地图，否则会削弱洞穴潜水题材最重要的不确定性。

### 5.3 水下交互统一原则

建议逐步收敛为：

- 一个主操作按钮
- 长按触发关键动作
- 用上下文决定当前动作语义
- 必要时通过工具轮盘承载次级功能

这样后续无论是铺绳、放标记、撤离、交互还是特殊剧情操作，都能落在同一套交互语言下。

### 5.4 结算 UI

每次返回岸上后的结算，不应只写"成功 / 失败"，而应强调：

- 这次带回了什么认知成果
- 哪些信息仍不确定
- 下一次最值得验证什么

建议至少呈现：

- 最深点
- 新区域数
- 新地标数
- 新增绳索与标记
- 返程原因
- 下一步建议目标

## 六、程序、美术、UI 三线协作接口

为了避免后续三条线各自为政，建议明确这些接口：

### 6.1 程序对美术的需求

- 空间母题枚举
- 区域材质枚举
- 节点标记图形语义
- 叙事物件的最小可识别版本
- 岸上地图图层规范

### 6.2 程序对 UI 的需求

- 岸上认知图图层结构
- 水下操作提示状态
- 结算摘要字段定义
- 正式救援确认流程
- 新发现高亮规则

### 6.3 美术与 UI 共同约束

- 同一地标在场景内、地图上、结算页中的识别语言应尽量一致
- 绳索、标记、危险区的图形语义应固定，不应在不同页面频繁变换
- 岸上与水下可以采用不同信息密度，但不应采用彼此冲突的符号系统

## 七、排期建议

### 7.1 第一阶段：先做闭环成立

- 跨下潜持久化
- 岸上认知图基础页
- 侦察 / 救援 两类下潜分流
- 探路撤离协议
- 返岸结算摘要

### 7.2 第二阶段：先做认知语言成立

- 节点标记系统
- 洞室命名与地标列表
- 场景辨识度强化
- 岸上地图的模糊 / 确定 区分

### 7.3 第三阶段：先做兑现感成立

- 正式救援前评估
- 带人撤离时的压力强化
- 路线回放
- 更强的岸上复盘表现

---

## 八、本卷最重要的结论

- **程序第一优先级不是加更多事件，而是先让探路成果可以跨下潜保留。**
- **美术第一优先级不是铺满细节，而是先建立可一句话记住的空间母题。**
- **UI 第一优先级不是增加按钮，而是把岸上整理与水下执行的信息密度明确分开。**
- **多次下潜闭环一旦成立，绳索、地标、标记、结算和正式救援才会真正互相抬价值。**
