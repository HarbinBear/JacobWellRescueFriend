# 雅各布井救援 —— 详细设计文档

## 一、文档目的

这份文档不是面向玩家，而是面向**后续接手项目的模型与开发者**。

它的目标是：

- 让新会话中的模型在**尽量少读源码**的前提下，快速建立对项目的整体认知。
- 说明这个项目**在做什么、为什么这样设计、主要系统如何协同**。
- 为后续修改玩法、修剧情、调数值、补内容、接管重构提供统一背景。
- 帮助模型区分哪些是**核心设计约束**，哪些只是当前实现方式。

这份文档偏重**游戏设计、系统关系、状态流转、玩法规则、剧情结构、世界结构**。
代码层模块职责、文件入口、函数分工、调用路径，应写入配套的 `code.md`。

---

## 二、项目概述

### 2.1 项目定位

**游戏名称**：雅各布井救援（JacobWellRescueFriend）

**平台**：微信小游戏（依赖 `wx` API 与小游戏运行环境）

**技术栈**：TypeScript + Canvas 2D

**类型定位**：

- 2D 俯视角洞穴潜水探索
- 强氛围恐怖叙事
- 资源管理与路径判断
- 轻量动作对抗
- 附带独立竞技场模式

### 2.2 核心体验目标

项目的核心不是“爽快战斗”，而是以下几种体验的组合：

- **未知深水的压迫感**：视野有限、地形狭窄、深处黑暗、信息不完整。
- **潜水风险管理**：氧气、氮气、方向感、撤离路线都构成持续压力。
- **叙事驱动探索**：不是随机下潜，而是为了寻找失踪的朋友“熊子”。
- **逐次深入的章节结构**：多次下潜、每次更深、更危险、更接近真相。
- **小规模系统复用**：主线偏叙事与生存，竞技场偏纯动作与生存挑战。

### 2.3 产品结构概览

本项目包含三个大层级：

1. **主线剧情模式**
   - 核心模式。
   - 以洞穴地图探索、剧情触发、队友/NPC 协作、恐怖演出为主。

2. **食人鱼纯享版竞技场**
   - 独立模式。
   - 以波次敌人、近战挥击、清场效率、存活时间为主。

3. **菜单与章节入口**
   - 负责切换主线、竞技场、章节选择。
   - 支持从指定剧情阶段进入游戏。

## 三、设计原则

### 3.1 第一原则：黑暗与受限感

洞穴潜水题材的关键不在大地图自由移动，而在**受限**：

- 视野受手电与遮挡限制。
- 路径不总是宽敞可退。
- 玩家不是超人，不能长期滞留深处。
- 关键事件通常在“快没把握”的时刻触发。

因此，很多系统都围绕“让玩家感到自己进入了不该进入的地方”而设计。

### 3.2 第二原则：剧情必须由行动承载

剧情不是脱离玩法的文本插播，而是尽量依附在：

- 进入某区域
- 靠近某地标
- 跟随某 NPC
- 被地形卡住
- 手电损坏
- 接近异常物体
- 选择撤退或坚持

也就是说，本项目中的剧情推进，本质上是**空间驱动的状态机**。

### 3.3 第三原则：主线与竞技场共享基础系统，但目标不同

共享的基础系统包括：

- 玩家移动
- 地图碰撞
- 相机与渲染
- 凶猛鱼实体与攻击表现
- UI 渲染框架

但两者设计目标不同：

- **主线**：强调氛围、悬念、演出、剧情阶段推进。
- **竞技场**：强调持续生存、清怪节奏、击杀反馈、轮次压力。

因此很多系统会有“双模态行为”，不能简单按单一游戏模式理解。

### 3.4 第四原则：项目优先可迭代而非抽象完美

这个项目明显带有快速迭代特征：

- 很多逻辑集中在状态对象与更新循环中。
- 剧情依靠 `stage` 与 `flags` 驱动。
- 某些玩法演出通过硬编码地标坐标触发。

这不是缺陷，而是当前阶段的**创作型原型结构**。后续接手时应优先维持行为正确，再考虑抽象重构。

## 四、世界观与叙事框架

### 4.1 故事前提

玩家的朋友“熊子”在雅各布井潜水时失踪。玩家为了寻找并救援熊子，进行多次下潜。

叙事基底来自现实洞穴潜水事故的恐惧感：

- 狭窄缝隙
- 坍塌风险
- 迷失方向
- 氧气耗尽
- 深水黑暗中的未知生物或幻觉感

整个游戏的情绪基调是：

- 克制
- 压迫
- 悲剧预感
- 带有恐怖演出
- 带有执念式救援主题

### 4.2 叙事推进方式

叙事主要由以下几类机制触发：

- **剧情阶段 `state.story.stage`**：主进度轴。
- **剧情标记 `state.story.flags`**：用于记录细粒度事件是否已经发生。
- **地标坐标触发**：例如潜水服、缝隙入口、手电损坏点、恐怖事件点。
- **NPC 行为**：熊子、潘子等角色的移动、跟随、救援行为。
- **玩家选择**：继续深入、上浮、长按放弃救援。

### 4.3 章节节奏

整个主线不是一次连续长流程，而是“多次下潜”的结构。这样设计有三个好处：

- 让剧情自然形成阶段性目标。
- 让玩家每次都能感受到“这次比上次更深”。
- 让失败、撤退、救援、返回水面都能成为叙事的一部分。

## 五、章节结构与剧情阶段

### 5.1 `stage` 是主线剧情的核心驱动轴

项目中主线剧情以 `state.story.stage` 为主导。模型接手时应优先把 `stage` 视作：

- 当前故事位于哪一章
- 哪些事件应该允许触发
- 哪些角色该出现
- 哪些 UI 应显示
- 哪些结局分支已开放

### 5.2 各章节语义

| 章节 | `stage` 范围 | 设计语义 |
|------|--------------|----------|
| 第一关 | 1~2 | 初次下潜，建立世界、建立目标、建立“进不去”的挫败感 |
| 第二关 | 3~6 | 二次下潜，真正进入危险区域，体验“被卡住濒死” |
| 第三关 | 7~8 | 独自深入，灯光异常、恐怖演出、放弃与坚持的道德压力 |
| 第四关 | 9 | 更深层的寻找与真相区域，当前仍在开发中 |

### 5.3 第一关：建立目标与阻隔

第一关设计任务：

- 告诉玩家为什么要下潜。
- 把熊子作为明确目标对象立起来。
- 让玩家第一次看到关键狭缝区域。
- 通过坍塌/阻挡机制明确告诉玩家：**你现在还进不去**。

关键作用不是完成救援，而是建立“必须再来一次”的动机。

### 5.4 第二关：把危险变成亲身体验

第二关的设计重点是将第一关“看到危险”，升级为“自己成为危险的受害者”：

- 玩家更深入。
- 进入狭窄通道。
- 被卡住。
- 氧气耗尽。
- 视野收窄与红屏加深。
- 最终由潘子救出。

这是主线里非常关键的一段，因为它不只是剧情文本，而是对洞穴潜水风险的**玩法式说明**。

### 5.5 第三关：主观恐怖与道德选择

第三关的设计气质和前两关不同，重点不是纯粹物理危险，而是：

- 独自行动带来的孤立感。
- 手电损坏造成的可靠信息源崩溃。
- 鱼眼闪现的惊吓演出。
- 灰色氧气罐与异常视觉事件。
- “长按放弃救援”的心理压迫。

第三关把“救援”从空间问题提升为**意志问题**。

### 5.6 第四关：深层目标区

第四关当前仍在开发中，但从现有地图与地标命名可以看出，它定位为：

- 更深层区域探索
- 接近最终真相或最终结果
- 真正意义上的终局区域

后续扩展时应尽量保持第三关之后的情绪承接，而不是突然转成普通探索关卡。

## 六、关键剧情事件设计

### 6.1 潜水服发现

潜水服是第一批强叙事地标之一，主要作用：

- 在视觉上提供“这里有人出过事”的证据。
- 为玩家建立环境叙事，而不仅是对白。
- 作为第一洞室的重要探索回报。

### 6.2 缝隙坍塌

这是第一关最关键的阻挡事件。

设计目的：

- 明确后续目标位置。
- 阻止玩家过早进入后段剧情。
- 让熊子“消失在你够不到的地方”。
- 形成心理债务：你已经看到他进去，但你没法立刻救他。

### 6.3 濒死体验

这是第二关的核心体验节点。

表现层通常包括：

- 移动受阻
- 氧气接近归零
- 屏幕红色叠加
- 视野收缩
- 压力持续上升
- NPC 救援介入

它是整个项目最能体现“潜水恐惧”的玩法桥段之一。

### 6.4 手电筒损坏

手电是玩家在黑暗中最重要的控制权来源，因此它损坏的效果不只是减弱光照，而是削弱玩家的心理安全感。

手电损坏后，系统层面会影响：

- 可见性
- 光照稳定性
- 对凶猛鱼的威慑效果
- 后续恐怖演出的可信度

### 6.5 鱼眼闪现

鱼眼演出属于典型的“短时高强度惊吓点”，设计上应理解为：

- 不是普通敌人出场
- 不是常规战斗触发
- 而是环境异常与深水恐惧的具象化表达

它要服务的是情绪峰值，而不是伤害数值。

### 6.6 放弃救援按钮

“长按放弃救援”是一个非常重要的设计装置，它的作用不是给玩家一个方便退出键，而是把“撤退”显性化、道德化。

其设计价值包括：

- 让玩家明确意识到自己在做选择。
- 让“放弃”变成需要主动承担的行为。
- 通过长按而非轻触，增加心理重量。
- 让结局分支具有参与感，而不是被动播放。

### 6.7 熊子死亡结局

当前至少存在两种可达成方式：

1. 长按放弃救援。
2. 第三关手电损坏后选择上岸。

在设计上，这两种本质都属于“未完成救援”的失败型叙事分支，只是心理含义不同：

- 主动放弃：更直白。
- 转身离开：更克制，但同样残酷。

## 七、游戏模式设计

### 7.1 主线剧情模式

主线模式由 `state.screen = 'play'` 表示。

它包含以下核心机制：

- 洞穴地图移动
- 氧气消耗与生存压力
- 氮气/深潜风险管理
- 光照与视野限制
- 剧情触发
- NPC 协作或跟随
- 引导绳部署
- 特定敌对/惊吓事件

主线模式最重要的不是“全系统都一直启用”，而是**根据章节阶段有选择地启用事件与规则**。

### 7.2 食人鱼纯享版竞技场

竞技场模式由 `state.screen = 'fishArena'` 表示，是一个从主线体系中分离出来的动作生存玩法。

特点：

- 无限氧气
- 无主线剧情推进
- 无复杂 NPC 同行逻辑
- 小而封闭的战斗地图
- 以波次敌人与清场节奏为中心

竞技场的存在意义：

- 复用凶猛鱼与攻击系统
- 提供一个与主线完全不同的即时反馈模式
- 增加项目可玩性与重复游玩价值
- 允许在不破坏主线节奏的前提下测试敌人与战斗手感

### 7.3 菜单系统

菜单以 `state.screen = 'menu'` 为总入口，并通过 `state.menuScreen` 区分子页面。

常见子页面：

- `main`：主菜单
- `chapter`：章节选择

菜单除了展示按钮外，还承担：

- 切换游戏模式
- 控制哪些入口可用
- 按项目阶段开放章节
- 在特殊模式下禁用主线入口

### 7.4 特殊模式开关

`fishArenaMode = true` 时，主线入口置灰，只开放竞技场。

这说明该项目存在一种“演示/限制模式”思路：

- 可以把版本裁成纯竞技场体验
- 可以临时禁用未准备好的主线内容
- 可以作为测试或发布策略的一部分

## 八、地图系统设计

## 8.1 主线地图基础规格

主线地图是固定洞穴结构，而非程序化随机图。

参数：

- 地图尺寸：80 列 × 150 行
- 单格大小：40 × 40 像素
- 总尺寸：3200 × 6000 像素

这种纵向长地图非常适合表达“向下深入”的心理感受。

### 8.2 地图数据来源

地图并非纯手写数组，而是通过素材图片处理生成：

- 原始视觉资源：`texture/Well.png`
- 处理脚本：`scripts/processMap.py`
- 产出数据：`src/world/mapData.ts`

这意味着地图编辑工作流更偏向：

- 先画轮廓或区域图
- 再由脚本转成游戏内可用数据

优点：

- 对关卡形状编辑直观
- 便于迭代整体洞穴结构
- 适合项目这种地形主导型玩法

### 8.3 地图格子语义

项目中至少有以下几种格子/碰撞语义：

- `0`：水域，可通行
- `1`：生成过程中的岩石标记，后续会转换
- `wall 对象`：边缘可视岩石，用于渲染圆润墙面与碰撞
- `2`：内部实体岩石，主要服务碰撞，通常不作为圆形边缘渲染

这个设计体现出：

- **渲染墙体**和**碰撞墙体**不是完全同一层抽象。
- 地图最终不是纯数字数组，而是混合数据结构。

### 8.4 区域划分 `state.zones`

区域划分不是装饰性命名，而是用来帮助理解：

- 当前玩家位于哪一段叙事空间
- 哪些剧情应在附近发生
- 哪些路径是主路、支路、死路

现有区域包括：

| 区域名 | 设计语义 |
|--------|----------|
| `entrance` | 水面入口与安全感残留区域 |
| `chamber1` | 早期探索区，较宽，作为进入深处前的缓冲 |
| `suit_tunnel` | 导向潜水服地标的狭窄通道 |
| `chamber2` | 更深入、更偏移的中段洞室 |
| `junction` | 路径决策区，连接多个方向 |
| `dead_end` | 用于误导、恐惧与 NPC 行动演出的假终点 |
| `chamber3` | 深层剧情区，承接第三关恐怖事件 |
| `story_tunnel` | 极窄、专门为剧情压迫感服务的通道 |
| `chamber4` | 终局潜在区域，当前开发中 |

### 8.5 关键地标 `state.landmarks`

地标是剧情与地图绑定的核心纽带。

| 地标 | 用途 |
|------|------|
| `entrance` | 入口出生点，通常自动扫描确定 |
| `suit` | 废弃潜水服位置，触发环境叙事 |
| `tunnelEntry` | 缝隙入口，第一关阻挡、第二关进入 |
| `tunnelEnd` | 玩家被卡住位置，濒死剧情核心点 |
| `deadEndDeep` | 假烟囱深处，服务第二关 NPC 行为 |
| `chamber12Junction` | 第一二洞室连接点，第三关灯损坏触发 |
| `chamber23Junction` | 二三洞室连接点，恐怖事件关键区域 |

### 8.6 地图设计意图

从区域与地标分布可以看出，主线地图并不是“大世界开放探索”，而是：

- 有一定自由度
- 但核心剧情路线上高度引导
- 关键事件强依赖具体坐标
- 空间结构服务情绪爬升

因此后续改图时要极其注意：

- 不要轻易改坏地标相对关系
- 不要让关键剧情点之间的路程节奏失真
- 不要把本应狭窄压迫的路径改成宽阔安全区

## 九、竞技场地图设计

### 9.1 竞技场基础规格

竞技场是独立正方形场地：

- 2000 × 2000 像素
- 格子大小 40 × 40
- 外围存在厚岩石墙

### 9.2 障碍物与空间目标

内部会随机生成矩形障碍物，且会避免离出生点太近。

作用：

- 提供绕行与拉扯空间
- 避免纯空场导致敌人行为单调
- 让玩家在被追逐时有可利用地形
- 让光照与近战攻击产生空间选择

### 9.3 竞技场设计语义

与主线不同，竞技场地图不需要讲故事，它要保证的是：

- 出生安全
- 敌人生成合理
- 路径可周旋
- 小地图信息清晰
- 节奏适合反复开局

因此竞技场全图默认已探索，是一个明确的“玩法场”而不是“叙事场”。

## 十、玩家核心生存系统

### 10.1 移动与朝向

玩家在俯视角下移动，朝向会影响：

- 手电照射方向
- 攻击方向
- 角色绘制方向
- 与敌人/场景的互动感受

在这种题材中，“朝向”不仅是动作参数，也是感知参数。

### 10.2 氧气系统

氧气是项目最基础的生存资源。

它承担的设计作用包括：

- 限制探索深度与停留时间
- 强迫玩家考虑回程
- 在狭窄地带制造焦虑
- 为濒死演出提供硬逻辑支撑

主线中氧气是压力来源，竞技场中则被解除，用于让战斗模式更纯粹。

### 10.3 氮气/深潜压力

虽然项目主打氧气管理，但氮气或深潜相关状态同样服务于“潜水不是简单血条”的概念。

它强调：

- 深度不是纯视觉变化，而是系统性风险变化。
- 玩家在深处承受的不仅是空间威胁，还有身体负担。

### 10.4 手电系统

手电是整个项目中最关键的感知工具之一，兼具：

- 照明
- 导航
- 氛围塑造
- 危险揭示
- 敌人威慑

它并不是简单的画面特效，而是直接影响玩法与情绪的主系统。

### 10.5 受伤、濒死与失败表现

项目中的失败表现通常会通过以下内容组合体现：

- 红屏叠加
- 屏幕震动
- 视野收缩
- 控制权下降
- 失败提示文本

这使得失败不只是一个布尔结果，而是一个逐步逼近的过程。

## 十一、引导绳系统设计

### 11.1 引导绳的题材意义

洞穴潜水中的引导绳非常关键，因此它在本项目里既有真实性，也有玩法价值。

它的意义在于：

- 体现专业潜水语境
- 帮助玩家建立路线感
- 在视觉上强化“我来过这里”
- 让深处探索更有仪式感

### 11.2 交互方式

当前绳索交互不是即时点击，而是结合：

- 靠近墙体锚点
- 玩家保持相对静止
- UI 提示出现
- 长按完成起始或结束固定

这种交互方式的设计价值：

- 防止高速移动时误触
- 让“布绳”显得更像一个实际操作动作
- 让锚定行为具有明确意图感

### 11.3 锚点选择规则

绳索系统会在玩家附近寻找合适墙体，并把绳端固定在墙体表面锚点，而不是墙心。

这意味着系统关心的是：

- 视觉合理性
- 绳索与墙体的贴合感
- 路径避障后的可读性

### 11.4 绳索路径

绳索不是简单直线，而会构建一条避开障碍的路径。

这很重要，因为洞穴地图大量存在狭窄墙体结构。若直接画直线，会造成：

- 穿墙
- 视觉不可信
- 玩家误判路线

因此引导绳实际上也是一个轻量路径生成系统的表现层应用。

### 11.5 绳索状态

从设计语义上看，绳索有几种关键状态：

- **未铺设**：等待起点锚定
- **进行中**：已固定起点，尾端跟随玩家
- **收紧中**：准备固定终点时的过渡状态
- **完成**：两端固定，形成稳定引导线

### 11.6 绳索的体验作用

引导绳除了“找路”外，还有几个额外效果：

- 提升洞穴探索的专业感
- 让玩家主动建立安全感
- 让深处场景更有可追溯性
- 为画面增加空间层次

## 十二、凶猛鱼与战斗系统设计

### 12.1 凶猛鱼的定位

凶猛鱼不是传统小怪，更接近一种：

- 水下危险生物
- 惊吓与压迫装置
- 带有行为模式的掠食者

在主线里，它服务氛围与局部危机；在竞技场里，它被提升为主要玩法对象。

### 12.2 行为状态机

凶猛鱼具备较细的 AI 状态：

- `roam`：自由游弋
- `detect`：发现玩家
- `stalk`：悄悄接近
- `circle`：环绕观察
- `lunge`：蓄力扑击
- `bite`：咬住玩家
- `devour`：吞食处理
- `retreat`：进食后撤离
- `fear` / `flee`：被灯光驱赶后的受惊与逃离
- `hit`：被打中或被逼退时快速撤离
- `dying`：死亡动画阶段

这套状态机说明敌人设计重点在“行为表演感”，不是简单追踪碰撞。

### 12.3 怕光机制

凶猛鱼会受到手电照射影响，但不是只要进入视野就逃跑，而是结合：

- 与玩家距离
- 在手电扇形范围内的位置
- 手电当前亮度
- 手电是否损坏闪烁

这使得手电既是照明工具，又是对敌手段。

### 12.4 扑击与吞食流程

凶猛鱼攻击有明显的节奏段落：

1. 观察或绕圈
2. 蓄力
3. 冲刺扑击
4. 咬住玩家
5. 吞食与死亡过场

这种设计让玩家有心理预警窗口，也让敌人更像捕猎者，而不是直接贴脸掉血源。

### 12.5 玩家攻击方式

玩家使用氧气瓶挥击进行近战攻击。

特点：

- 扇形判定
- 有攻击角度与范围
- 有冷却
- 有挥击表现与停留残影
- 命中后会触发明显反馈

当前敌鱼设计偏“高机动但脆弱”：常态下很难近身，但一旦被真正打中，就会进入死亡流程。

### 12.6 竞技场中的战斗定位

在竞技场里，这套系统从局部事件转化为主循环：

- 生成敌鱼
- 周旋走位
- 借障碍规避
- 抓扑击间隙反打
- 清完一波后进入下一波

因此，竞技场其实是凶猛鱼 AI 与玩家攻击系统的集中展示场。

## 十三、NPC 与同伴设计

### 13.1 NPC 的主要职责

主线中的 NPC 不是纯对话对象，而是承担以下作用：

- 叙事推进
- 目标引导
- 陪伴感与对照感
- 关键救援行为
- 路线示范或误导

### 13.2 熊子

熊子是失踪者，也是驱动主线全部行动的中心角色。

他的设计功能包括：

- 作为被寻找目标提供明确任务动机
- 通过“先一步进入危险区域”制造追逐感
- 通过无法立刻救出形成持续悬念
- 通过可能死亡结局承担悲剧重心

### 13.3 潘子

潘子在第二关中的核心功能是“把危险的不可逆，暂时转回可逆”。

也就是说，他不是一般意义上的陪同 NPC，而是：

- 见证二次下潜
- 参与危险更深层区域
- 在玩家濒死时完成救援
- 让第二关形成结构上的闭环

### 13.4 NPC 的设计原则

NPC 不应喧宾夺主。

他们存在的目的不是持续占据画面中心，而是：

- 在关键时刻推动情绪变化
- 让玩家行为更有叙事对照
- 让“独自下潜”和“有人同行”的差异更明显

## 十四、输入与交互设计

### 14.1 输入目标

项目面向微信小游戏，因此输入设计必须兼顾：

- 触摸操作
- 虚拟摇杆或触控区域
- 菜单点击
- 长按确认
- 可能存在的键盘调试支持

### 14.2 交互分层

可以把交互理解为三层：

1. **移动控制**：持续输入，影响角色位移与朝向
2. **动作输入**：攻击、确认、长按等瞬时行为
3. **界面输入**：菜单切换、章节选择、模式切换

### 14.3 长按交互的重要性

项目中长按不只是一个输入技巧，而是用于强调“需要决心”的操作，例如：

- 铺设绳索
- 放弃救援

所以后续新增关键选择时，优先考虑是否也该使用长按，而不是轻触。

## 十五、UI 与反馈设计

### 15.1 UI 不是单纯信息栏

本项目的 UI 除了展示数值，还承担：

- 引导玩家当前该做什么
- 告诉玩家事件是否可触发
- 表现危险程度
- 承接菜单与章节入口
- 呈现剧情选择与结局态

### 15.2 主要 UI 职责

现有 UI 系统至少覆盖以下内容：

- 主 HUD
- 菜单页面
- 操作说明页
- 绳索长按按钮/进度提示
- 攻击挥击特效
- 结算或失败信息
- 剧情相关提示语

### 15.3 视觉反馈类型

项目大量依赖即时反馈来补足简化美术条件下的表现力，例如：

- 红屏
- 震动
- 图标提示（问号、感叹号）
- 按钮进度圈
- 闪烁灯光
- 攻击残影

这些反馈是项目体验的重要组成部分，不能轻易删减。

## 十六、渲染与氛围设计

### 16.1 渲染目标

Canvas 2D 渲染在这个项目里不是为了复杂 3D 效果，而是为了实现：

- 手工控制的黑暗空间感
- 稳定的俯视角表现
- 可定制的光照遮罩
- 低成本但强氛围的演出

### 16.2 光照是氛围主轴

光照系统至少承担：

- 生成手电可视区域
- 提供遮挡后的不确定性
- 与泥沙、雾感、黑暗叠加
- 为凶猛鱼怕光机制提供玩法依据

### 16.3 角色与敌人绘制

角色和敌人都采用风格化绘制方式，而非复杂逐帧贴图。

其优点：

- 便于程序驱动状态表现
- 便于依据角度、状态、动画时间做细微变化
- 便于快速调整体型与反馈

### 16.4 绳索与环境层

绳索、墙体、洞穴边缘、光照遮罩共同组成画面的空间可信度。

在这种黑暗潜水题材中，哪怕只是多一条绳索路径，也会显著提升“这个空间有人真实穿行过”的感觉。

## 十七、状态管理设计

### 17.1 全局状态树是项目中枢

项目以集中式状态对象驱动。

这意味着：

- 大部分系统通过共享状态通信
- 渲染与逻辑依赖同一批状态数据
- 模式切换实质上是状态切换
- 剧情推进实质上是状态字段变化

### 17.2 需要重点关注的状态域

后续接手时，应优先理解这些状态分区：

- **屏幕/菜单状态**：当前处于主菜单、主线、竞技场、结局还是失败
- **玩家状态**：位置、速度、朝向、生存参数、动作状态
- **剧情状态**：`stage`、剧情文本、演出参数、剧情标记
- **地图状态**：地图数据、墙体、区域、地标、探索信息
- **绳索状态**：当前铺设中绳索、已完成绳索、UI 进度、长按信息
- **敌人状态**：凶猛鱼列表、被咬流程、攻击动画
- **竞技场状态**：波次、计时、击杀、结算相关

### 17.3 状态驱动的优点与风险

优点：

- 迭代快
- 事件联动直观
- 适合原型期密集创作

风险：

- 状态字段增多后容易相互耦合
- 新剧情分支可能遗忘清理旧状态
- 多模式共存时容易出现边界污染

因此后续改动时要特别注意重置逻辑与模式切换逻辑。

## 十八、主循环与模式切换设计

### 18.1 启动流程

项目入口大致遵循以下流程：

- 初始化纹理
- 注册输入
- 重置游戏逻辑
- 进入持续更新与绘制循环

### 18.2 双更新结构

主循环中至少存在两套逻辑更新语义：

- `update()`：主线相关逻辑
- `updateArena()`：竞技场相关逻辑
- `draw()`：统一渲染出口

这说明该项目不是把竞技场做成完全独立子工程，而是通过同一主循环分流不同模式。

### 18.3 屏幕状态流转

可将 `state.screen` 理解为项目最外层状态机：

```text
menu -> play -> ending / lose
menu -> fishArena -> lose / menu
```

实际项目中还会叠加章节选择、结局分页、模式返回等细分流程。

## 十九、结局与失败设计

### 19.1 失败不等于游戏结束语义

在这个项目中，“失败”有两种含义：

1. **系统失败**：例如被凶猛鱼击杀、氧气耗尽。
2. **叙事失败**：例如放弃救援、回到水面导致熊子死亡。

两者都可能结束当前流程，但情绪含义不同。

### 19.2 结局的设计价值

项目中的结局不是单纯奖励，而是为了强化主题：

- 救援是否值得继续
- 你能承受多深的风险
- 你是否真的敢面对洞穴深处
- 有些下潜即使完成，也未必是真正意义上的胜利

### 19.3 第三关分支的重要性

第三关是当前最明确的选择分支关卡，因此后续新增分支时，应尽量遵循它的设计原则：

- 决策应嵌入情绪高压场景
- 决策应有动作成本
- 决策结果应具有叙事后果

## 二十、工程与资源设计约束

### 20.1 工程依赖保持轻量

当前工程依赖非常轻，核心开发依赖只有 TypeScript。

这说明项目倾向于：

- 原生 Canvas 逻辑
- 少框架、少封装
- 更直接地控制运行时行为

### 20.2 资源目录语义

从目录结构可以推断资源分层大致如下：

- `audio/`：音频资源
- `texture/`：贴图与地图源图
- `scripts/`：构建或素材处理脚本
- `src/`：核心源码
- `typings/`：类型声明
- `.codebuddy/rules/`：供模型理解项目的规则文档

### 20.3 当前阶段的合理认知

这是一个**正在持续扩写内容的单机型创作项目**，不是已经完全工程化的商业大项目。

因此接手时的优先级建议是：

1. 先理解现有行为与设计意图
2. 再补内容或修问题
3. 最后再考虑抽象和重构

## 二十一、后续扩展建议

### 21.1 扩展剧情时

应优先复用现有结构：

- 继续沿用 `stage`
- 继续沿用地标触发
- 继续沿用 `flags` 记录细节状态
- 尽量让剧情依附在移动、到达、等待、长按、撤退等动作上

### 21.2 扩展地图时

需要优先保证：

- 新区域的情绪作用清晰
- 地标相对关系不被破坏
- 狭窄/宽阔节奏仍可控制
- 光照与撤退路线仍有意义

### 21.3 扩展敌人与战斗时

不应把项目带偏成纯动作游戏。新增战斗内容应继续服务：

- 氛围
- 生存压力
- 深水未知感
- 特定模式下的动作挑战

### 21.4 扩展 UI 时

优先保持：

- 可读性强
- 情绪一致
- 黑暗水下主题一致
- 不要让 HUD 过满破坏沉浸感

---

## 二十二、给后续模型的理解优先级

如果是新会话第一次接手这个项目，建议按以下顺序理解：

1. **先理解这是一个“洞穴潜水救援叙事游戏”**，不是普通动作游戏。
2. **再理解 `state.story.stage` 是主线进度主轴**。
3. **再理解地图区域与地标决定剧情发生位置**。
4. **再理解光照、氧气、绳索共同构成探索压力**。
5. **最后再理解竞技场是对敌人与战斗系统的独立玩法化复用**。

如果忽略以上顺序，容易把局部实现误读成项目核心。

---

## 二十三、最重要的项目认知摘要

可以把整个项目浓缩成一句话：

**这是一个以洞穴潜水压迫感为核心、以多次下潜救援失踪朋友为叙事骨架、以光照/氧气/地形/剧情触发为主要系统支柱，并附带独立凶猛鱼竞技场模式的微信小游戏。**

如果再进一步拆成四点：

- **主线核心**：救援、下潜、深入、恐惧、选择。
- **系统核心**：状态机、地标触发、光照、生存资源、绳索。
- **情绪核心**：黑暗、狭窄、未知、濒死、放弃的重量。
- **扩展核心**：在不破坏主线氛围的前提下复用系统做竞技场与新章节。
